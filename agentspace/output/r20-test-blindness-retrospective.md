# 深度反思：为什么「修类不修实例」清单入册的同一轮，修复自己还是留下了同族漏网（r20 F-1 / F-2）

- 关联：`deep-review-2026-07-11-r20.md`（问题发现与修复）、`r17-/r18-/r19-test-blindness-retrospective.md`（前三轮复盘）

## 〇、先给结论

r20 的两个头部致命项都不是新家族——它们分别是 **r19 两个修复的直系亲属**：

1. **F-1（between 引用不入 JOIN 树）是 r19 F-2 的兄弟**。r19 修「EXIST 载荷里的 isReferenceValue 引用要并入外层 JOIN 树」时，新写的收集器 `collectExistReferencePaths` 原样复制了 r12 修复的字符串识别（`typeof value[1] === 'string'`）——而 `getFinalFieldValue` 里 between 分支明明白白对 `value[1][0]`/`value[1][1]` 做引用编译。**同一个文件里，编译器认识两种引用形态，入树逻辑只认识一种**，且这个不一致被从 r12 原样抄写到 r19 的新代码里。
2. **F-2（行内写路径 × filtered 视图整族缺失）是 r19 F-3 的家族本体**。r19 修的是「combined 抢夺时宿主的 filtered entity 缺 delete 事件」——修复用 `collectMembershipChecks(宿主名, …)` 精准补了**宿主**这一个消费者。但病灶不是「宿主漏了快照」，而是「**手工 push 事件的行内记录（merged link / combined 记录）整个绕过了记录级钩子**」。以行内记录自身为 base 的视图（filtered relation、combined 记录上的 filtered entity）在全部七种行内写法下事件为零。r19 修复恰好选了那个**不在**行内记录自身视图上的复现（宿主侧视图），于是家族里其余七个格安然无恙地活到 r20。

这两个漏网发生在把「修类不修实例」写进 AGENTS.md 强制清单（r18）之后的第一轮修复里。清单存在，但没有被机械地执行——原因值得挖。

---

## 一、显微镜：清单为什么没拦住

### 1.1 「枚举全部读者」在什么维度上枚举，决定了能不能看见兄弟

r19 修 F-2 时确实做了读者枚举——报告原文："r12-F-2 修了外层 direct match 的引用路径，EXIST 载荷是同族漏网读者"。枚举的维度是**引用出现的位置**（direct match / EXIST 载荷）。但引用声明面还有第二根维度：**引用的值形态**（字符串 / between 引用对）。位置维度枚举完两个值，就宣布收口了；形态维度从未被当成维度。

**教训的一般形**：声明面的读者枚举必须先问「这个声明面有几根正交轴」，再做轴上取值的笛卡尔枚举。只沿一根轴枚举，其他轴上的兄弟全部不可见。r17 复盘说的「维度登记册」机制本来就是干这个的——但登记册当时只登记**测试矩阵**的维度，没有要求**修复时的读者枚举**也对着登记册做。（本轮把「引用形态」轴回灌登记册，并把收集器收口为唯一登记点，见 §3。）

### 1.2 「汇合点收口」被字面执行成了「在我修的分支上建汇合点」

r19 F-3 的修复其实建了一个小汇合点（复用 collect/settle 机制），但汇合点建在「抢夺场景的宿主侧」——它汇合的是 r19 复现走过的那一条路径。真正的汇合点应该问的是：**这类事件是从哪里产生的？** 答案是「手工 push」——`preprocessSameRowData`、`flashOut`、`DeletionExecutor` 三处共约七个 push 点，全部绕开记录级钩子。「手工 push 的事件」才是类；「combined 抢夺」只是实例。
本轮修复把汇合点建在正确的层级：`FilteredEntityManager` 的 post-write 任务队列 + 删除快照扩展——任何手工 push 的行内记录事件都必须伴随视图成员资格任务，写入完成点统一 drain。今后新增手工 push 点时，遗漏 enqueue 依然可能（见 §4 有意不做），但至少机制存在、样板统一。

### 1.3 结构性测试层的盲区：矩阵有「拓扑轴」也有「视图轴」，但没有它们的交叉格

r17 建立的 `writePathTopologyMatrix` 覆盖拓扑 × 操作，r16/r18 的 `filteredMembershipMatrix` 覆盖视图 × 谓词形态——**两张矩阵各自完备，交叉为空**（r18 报告第三节第 6 条其实已点名「物理拓扑 × filtered entity 交叉格空白」，但被排为"防回归性价比"项而非"必然藏着致命 bug"项，一直没铺）。F-2 的七个格全部住在这个交叉里。
**教训**：当两根轴的矩阵各自存在时，「交叉格空白」不是欠一点覆盖率，而是欠一整个家族的观测面。已点名的空白格应当按「这里有 bug，只是还没复现」处置，优先于新矩阵的开辟。

## 二、这两个 bug 的公共形状

> **修复自身也是「声明面的一个读者」。** 修 bug 的代码继承了它所修代码的形态假设（F-1 抄了字符串判断）、或者只覆盖了复现走过的消费者（F-2 只补宿主侧）。「修类不修实例」失败的主要方式不是没听说过这条规则，而是**把「类」错误地定义成了「我复现出来的那一格」**。

判定「类」的两个可操作检验（本轮回灌 AGENTS.md 清单的执行细则）：

1. **形态检验**：修复引入/复制的每一个类型判断（`typeof x === 'string'`、`Array.isArray`、`instanceof`），都要问「这个声明面还有哪些合法形态没进这个分支」。编译/求值侧认识的形态集合与守卫/入树侧认识的形态集合必须相等——不等就是漏网清单。
2. **产生点检验**：修复某个事件/数据缺失时，先枚举「这类事件的全部产生点」（grep push 点/写点），病灶定义为「绕过共同管线的产生点集合」，而不是「出问题的那个场景」。

## 三、本轮已落地的机制改造

1. **引用形态唯一登记点**：`MatchExp.collectAtomReferencePaths` 成为 isReferenceValue 引用形态（字符串 / between 对）的唯一识别器，direct match 入树与 EXIST 收集两个消费方共用。新增引用形态只改一处，两个消费方自动同步。
2. **行内视图成员资格汇合点**：`FilteredEntityManager.enqueuePostWriteCreationCheck` / `enqueuePostWriteUpdateCheck` / `settlePostWriteChecks`（挂在 events 数组、写入完成点统一 drain）+ `collectInlineDeletionSnapshot` + `collectDeletionMemberships` 的行内 link 子快照。手工 push 的行内事件从此有配套的视图事件产生机制。
3. **事件模式匹配汇合点**：`mergedMutationEventView` 由 TransitionFinder 与 eventDep 匹配器共用（F-5 的收口），RecordMutationEventPattern 的 update 匹配语义只有一份实现。
4. **登记册回灌三根轴**：`WritingComputationTests.md` 新增「视图 × 写形态交叉格」「事件模式匹配语义」「引用形态」三行，并各自标注唯一登记点/汇合点——今后修这些面的 bug 时，登记册直接给出兄弟格清单。

---

## 四、有意不做（含理由）

- **不做「手工 push 事件必须带视图任务」的运行期强制**：手工 push 点与 enqueue 点是一对多的松耦合（delete 形态在 push 前快照、create 形态在 push 后求值），一刀切的运行期断言需要给事件打产生点标记，侵入所有 push 点，成本高于收益。以登记册交叉格 + 事件完备性预言机（矩阵已接入）作为检测面。
- **不把 updateRelationByName 的端点断言与 UpdateExecutor 守卫合并成一处**：前者是 API 面（全量拒绝，含 null 清除歧义形态），后者是执行面（幂等引用放行，Transform patch 依赖）。两个面的容许集合本就不同，强行合并会把执行面的幂等豁免泄漏到 API 面。契约（不可重指）一致即可。

---

## 五、一句话总结

**r17 说坐标系缺数据形态轴，r18 说同一声明面的每个读者都是一根轴，r19 说轴还长在数据流上游的原语里——r20 的教训是：修复代码自己就是声明面的又一个读者。「类」的边界不由复现定义，由声明面的正交轴笛卡尔积定义；修复时的读者枚举必须对着登记册做，修完必须把新发现的轴灌回去，否则登记册只保护测试、不保护修复。**
