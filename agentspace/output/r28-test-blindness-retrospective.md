# 深度反思：r28 的致命家族为什么在 r27 的防御升级后仍然存在

- 日期：2026-07-14
- 关联：`deep-review-2026-07-14-r28.md`、`quality-foundation-plan-r27.md`
- 性质：测试与修复体系的结构性复盘。本轮特殊性：这是**支柱 I（生成式测试）落地后的第一个完整周期**，四个致命家族全部由 fuzzer 种子先行抓获——复盘的对象因此从「为什么测试没抓到」转变为「为什么防御体系此前没抓到 + fuzzer 抓到它们需要什么条件」。

---

## 〇、先给结论：本轮验证并精化了 r27 的两个论断

r27 复盘的核心论断是：（1）缺轴无法靠清单补，轴的先验来源是实现分支点；（2）生成侧是瓶颈，预言机已经足够强。r28 的证据：

- **全部四个致命家族的预言机早已存在**（事件完备性 7 条规则、逻辑 id 唯一、双向一致），r27b 落地它们时一行未改就能判定 r28 的全部损坏——**再次证明缺的从来不是断言强度，是输入形状**。
- 但 G/H 家族在 r27b 首跑（种子 1–8 + 100–199 的 40 操作）只露出了一部分；**把种子池扩到 499、操作数扩到 50 之后才全部现身**。生成式测试不是一次性工程——**生成域的每一次扩张（更多种子、更长序列、更宽 schema 菜单）都等价于一轮新的人工审查**，而且成本低一个数量级。

## 一、F-1/F-3 的显微镜：编码手段被当作事实本身（本轮最深的一课）

### 1.1 错误的形状

`mergedTo='combined'` 的设计语义是「**用同一物理行编码这条配对**」。正确的蕴含方向只有一个：

> 配对存在 ⇒ 两记录同行（且 link id 列非空）。

实现却在几十个消费点上默认了逆命题：**同行 ⇒ 配对**。查询编译（combined x:1 无 JOIN 的同行读取）、删除级联（`record[relianceAttr]?.id` 判存在）、行删除足迹（静态类型足迹）全部按逆命题消费。只要运行期能装配出「同住但未配对」的行（hub 亡故余留、多 owner reliance 领养、F-1 的双 combined 放置），每个消费点就是一个静默损坏族。

### 1.2 为什么 27 轮没有系统性地暴露它

1. **逆命题在「单配对宇宙」里恒真**。绝大多数测试拓扑一张表只有一条 combined 关系——此时同行与配对确实一一对应，逆命题不可证伪。只有 schema 里出现**两条以上能把记录放进同一行的机制**（reliance × mergeLinks、多 owner reliance、hub 链），两个概念才分离。fuzzer 的 schema 生成器随机组合关系菜单，天然铺出了这类拓扑——这是「生成器铺格子」对「人肉枚举」的又一次结构性胜利：**人不会去测自己心智模型里不区分的两个概念**。
2. **真相源（link id 列）从未被登记为契约**。`findRelationByName` 一直按 link id 判定（正确面），嵌套读取按同行判定（错误面）——同一个「配对存在吗」问题在两个消费点有两个答案，而没有任何断言要求它们一致。这是 r27「声明-实现一致性」支柱的一个新变体：**同一语义谓词的多个实现必须对账**（r19 的 SQL NOT / 内存 not 对账是先例，本轮是「配对判定」）。
3. **F-1 的 Setup 早退把非法状态的制造成本降到了零**。`joinTables` 对「已同表」返回 undefined（成功语义）——双 combined 放置、互 reliance 全部静默通过声明期。r23 的同向重复守卫只覆盖了 (source,target) 完全相同的格——**守卫的键空间比不变量的键空间窄**（不变量是「实体对至多一条 combined 放置」，守卫只查了「同向同对至多一条 reliance」）。

### 1.3 机制化落地

- 行槽位排他成为 Setup 期不变量（`combineRecordTable` 的 co-located 检测——检测点就在早退处，非法状态从源头不可制造）；
- 配对真相源统一到 link id 列，消费面全枚举收口（读取剪枝 / match 守卫原子 / 删除核验 / 搬迁豁免标记 `physicalRowMatch`——**豁免必须显式**，flashOut 的物理寻址是特性不是漏网）；
- 登记册新增「同住 ≠ 配对」轴（取值 = 每个按同行消费配对事实的代码点）。

## 二、F-2 的显微镜：机制复用继承了错误语义

搬迁（relocate/flashOut 清旧行）复用 `deleteRecordSameRowData`，于是继承了逻辑删除的三个语义：级联（子树成员的独立表结构随"死亡"清理）、默认值补齐（缺席键 ⇒ defaultValue）、发号（link id 当新建分配）。三个都对删除正确、对搬迁全错。

**判定复用是否合法的问题不是「代码形状相同吗」，而是「调用方的语义前提同构吗」。** 删除的前提是「这些记录的逻辑生命结束了」；搬迁的前提是「这些记录换了物理住址」。前提不同构，复用就必须拆分（本轮的 `clearRowDataForMigration`）或参数化语义。同族先例：r22 F-2（`runWithTransactionRetry` 的两个消费方对 attempt 隔离的前提不同）。登记册新增「物理搬迁 ≠ 逻辑删除」轴。

另一个子教训藏在默认值改写里：**查询快照对 NULL 列省略键 + 写路径对缺席键应用默认值**，两个各自合理的约定在「快照回写」组合下变成静默改写。r21 F-1 已经把「键缺席 ⟺ 库里 NULL」登记为快照完备性契约——但那是**读侧**（本地谓词求值）的登记；本轮暴露的是**写侧**消费同一形状时的同一歧义。契约登记时应同时枚举读写两侧的消费方。

## 三、F-4 与 I-2：r27 F-2 的轴写窄了

r27 把「单事件 × 多监听扇出」轴登记为「property dep 合并 / records dep 独立」——**轴的取值枚举漏了事件驱动轨（eventDeps）和同源 records dep 的退化格**。r27 复盘操作性规则第 2 条（"每 X 注册一个 Y 必须回答多 Y 同时命中执行几次"）本身是完备的，但登记册回填时只写入了当轮修过的取值。教训：**回填登记册时按规则的全称量词展开取值，而不是按本轮 bug 的实例展开**。判定标准依旧是 r27 的「用户层可否去重」：Transform callback 收到相同事件对象且无 dep 身份 ⇒ 框架合并；Custom 的 planIncremental 有 `context.depKey` ⇒ 用户可分流 ⇒ 声明期强制显式化而非静默 N 倍。

## 四、I-1 的显微镜：拓扑等价审计应当是固定维度

r27 F-1 确立了「物理拓扑不得改变声明面合法性」（载荷形态 × 拓扑求全积）。本轮 I-1 是同一原理在**关系语义层**的投影：reliance 生命周期契约（re-parent/displacement/领养/幂等回写）在 combined 上有完整实现与 r27b 回归，在 merged/isolated 上六种形态全漏。**同一根轴在不同语义层反复咬人，说明它不应按语义层逐个回填，而应升格为固定审计维度**：任何「声明面语义 × 物理编译」的组合，在语义定义时就求全积。本轮的 reliance 拓扑等价矩阵（8 轨 × 3 拓扑探针）就是该维度的可复用样板。

## 五、本轮已落地的机制改造清单

1. Setup 期行槽位排他不变量（co-located 检测 + mergeLinks fail-fast + reliance 降级）；
2. 配对真相源收口（synthetic `&` + 读取剪枝 + match 守卫原子 + 删除核验补载 + `physicalRowMatch/physicalRowRead` 显式物理寻址标记）；
3. `clearRowDataForMigration`（搬迁与删除的机制分离）+ NULL 物化 + 携带 link id 保号 + relocate 端点翻转；
4. 搬运子树级 co-tenant 守卫统一（host-attr 轨接线 + `collectOutOfSubtreeCombinedPairings` 子树谓词）；
5. reliance 槽位守卫收敛于 `createRecord` 入口 + 方向感知的 owner 解除；
6. 事件驱动计算 (computation, event) 去重（live + migration 两消费方）；
7. Custom 同源多 dep 声明期 fail-fast；
8. clone(deep) 家族对齐（BoolExpressionData/Conditions/Activity/ActivityGroup）；
9. 登记册三根新轴/扩展（扇出轴取值补全、物理搬迁≠逻辑删除、同住≠配对）。

## 六、给后续轮次的操作性规则（增补）

1. **编码手段 ⇏ 事实**：任何「用物理布局编码逻辑事实」的设计（同行=配对、列非空=存在、行序=顺序……），必须显式登记事实的**唯一真相源列**，并枚举全部消费点验证它们读同一真相源；物理寻址的消费点（搬迁机制）必须**显式豁免**而不是默认混用。
2. **复用前判语义前提**：复用一段带副作用的例程前，回答「两个调用方对这些副作用的语义前提是否同构」——级联、默认值、发号、事件都算副作用。
3. **登记册回填按全称量词展开**：新轴入册时枚举规则的全部取值（含未咬人的），而不是只登记本轮实例。
4. **拓扑等价是固定审计维度**：每个声明面语义（合法性、生命周期契约、事件契约）在定义时即对全部物理编译（combined/merged/isolated × 各轨道）求全积；reliance 拓扑等价矩阵是样板。
5. **生成域扩张即审查**：每轮 review 至少做一次 fuzzer 生成域扩张（种子池 / 操作数 / schema 菜单三选一）并收口新红种子；固定 CI 种子集只防倒退，不产生新知识。
6. **组合契约双侧登记**：登记「形状 ⟺ 语义」类契约（如「缺席键 ⟺ NULL」）时，读侧与写侧的消费方同时枚举。
