# 深度反思：为什么已有的测试体系此前没有抓到 r28 的这些致命 bug

- 日期：2026-07-14（2026-07-15 深化修订）
- 关联：`deep-review-2026-07-14-r28.md`、`quality-foundation-plan-r27.md`、`r27-test-blindness-retrospective.md`
- 性质：测试与修复体系的结构性复盘。本轮特殊性：这是**支柱 I（生成式测试）落地后的第一个完整周期**——复盘对象因此从单一的「为什么测试没抓到」拆成三个更诚实的问题：（A）为什么**人写的测试体系**对这些家族结构性失明；（B）fuzzer 在 r27b 就已存在，为什么**现在**才抓全；（C）本轮有哪些致命项**连 fuzzer 也没抓到**、是靠什么抓到的。

---

## 〇、先给结论：三个问题的一句话答案

**（A）人写的体系为什么失明**——不是格子漏铺，而是三种更深的结构性原因：
1. **测试宇宙的受限性使两个不同概念在其中外延相等**（「同住」与「配对」在单配对拓扑里逐点重合，任何断言都无法区分它们）；
2. **夹具最小化 + 断言聚焦偏置**（测试只断言操作焦点上的变化，从不断言"世界的其余部分没变"——而这正是 F-2 全家族的藏身处）；
3. **用户本体论的公理从不被手写断言**（"id 不会变"、"无关记录不受影响"、"删除不会复活"——公理性质只有闭世界预言机会检查）。

**（B）fuzzer 为什么现在才抓全**——生成域的经济学：G 家族在 r27b 的 100 种子首跑就已现身并**被有意排队**（红-绿纪律，见 quality-plan §1.4——这部分不是失明，是「检出→修复」的时延）；H 家族（200–499 号种子、50 操作长序列）需要**更深的状态历史**才能装配出损坏前置状态——种子池/操作数的每一次扩张都等价于一轮新审查。

**（C）fuzzer 也没抓到的**——运行时扇出双跑（Transform eventDeps / Custom 同源 dep：fuzzer 不生成计算声明）靠**主动轴审计**（r27 规则的执行）抓到；查询面幻影读取靠**家族分析后的人工探针**抓到（fuzzer 的预言机缺「配对读取一致性」这条规则）。三种侦测机制各有辖区，本轮首次同时开工。

## 一、F-1/F-3 的法医检视：概念在测试宇宙中的外延坍缩（本轮最深的一课）

### 1.1 错误的形状

`mergedTo='combined'` 的设计语义是「**用同一物理行编码这条配对**」。正确的蕴含只有一个方向：

> 配对存在 ⇒ 两记录同行（且 link id 列非空）。

实现却在几十个消费点上默认了逆命题：**同行 ⇒ 配对**。查询编译（combined x:1 无 JOIN 的同行读取）、删除级联（`record[relianceAttr]?.id` 判存在）、行删除足迹（静态类型足迹）、`joinTables` 的「已同表=成功」早退，全部按逆命题消费。

### 1.2 防线逐一检视

| 防线 | 为什么没响 |
|------|-----------|
| 拓扑矩阵（r17 起，combined/merged/isolated 全铺） | 每个格子的 schema **一张表只有一条 combined 关系**。在这个宇宙里「同住 ⟺ 配对」逐点成立——逆命题**不可证伪**。要让两个概念分离，schema 必须有**两条以上把记录放进同一行的机制**（双 combined 放置、多 owner reliance、hub 链）——这类拓扑从未被人写出 |
| r23 F-1 守卫（重复 reliance 合表 fail-fast） | **守卫的键空间比不变量的键空间窄**：不变量是「实体对至多一条 combined 放置（含经链间接）」，守卫只查了「同向同 (source,target) 至多一条 reliance」。r23 执行了清单第 3 条（规则升级为守卫），但守卫锚定在**咬人的实例**上而不是实例背后的不变量上——写守卫时没有问「这条守卫是哪个不变量的影子」 |
| 声明期校验（validateCreateArgs 体系） | 互为 reliance（A⇄B）、reliance×mergeLinks 同对——这些形状**逐条声明都合法**，非法性只在组合里涌现。而且它们在人的心智模型里是"没道理的写法"——**人不会测试自己认为无意义的形状**（r27 已把这条命名为夹具偏置，本轮再次验证：taboo 形状恰恰是 fuzzer 的生成器无禁忌覆盖的地方） |
| 双实现对账（r19 先例：SQL NOT vs 内存 not） | 「配对存在吗」这个语义谓词有两个实现：`findRelationByName`（按 link id，正确）与嵌套读取（按同行，错误）。**同一谓词的双实现从未被要求互相对账**——r19 的教训登记在"布尔原语"上，没有泛化为「一切双实现语义谓词都需要一致性预言机」 |
| 行覆盖率 | `joinTables` 早退行、同行读取路径全部被绿测试**执行过**——覆盖率度量执行，不度量语义激发。（r24 起反复确认，不再展开） |

### 1.3 为什么 fuzzer 能抓到

schema 生成器从关系菜单随机抽样组合，**没有"这个组合没道理"的先验**——双 combined 对、互 reliance、hub 链在数百种子里必然出现。预言机（事件完备性 + 逻辑 id 唯一 + 双向一致）一行未改就把六种损坏全部判红。这是 r27 论断（缺的不是断言强度，是输入形状）的第二次实证。

## 二、F-2 的法医检视：夹具最小化与框架问题（frame problem）

### 2.1 机制复用继承了错误语义

搬迁（relocate/flashOut 清旧行）复用 `deleteRecordSameRowData`，继承了逻辑删除的三个语义：级联（子树成员的独立表结构随"死亡"清理）、默认值补齐（缺席键 ⇒ defaultValue）、发号（link id 当新建分配）。三个对删除全对、对搬迁全错。**判定复用是否合法的问题不是「代码形状相同吗」，而是「调用方对副作用的语义前提同构吗」**——删除的前提是"逻辑生命结束"，搬迁的前提是"换物理住址"。同族先例：r22 F-2（`runWithTransactionRetry` 两个消费方的 attempt 隔离前提不同）。

### 2.2 为什么既有的 relocate 测试全绿

`combinedRecordEvents.spec.ts` 的 relocate 用例（r17 落地，事件语义契约的正式载体）夹具是 **User–Profile 二元最小形态**：被搬的 Profile 没有 isolated link、没有显式置 null 的带默认值字段、没有 merged FK、行上没有 co-tenant。断言集是：link delete 事件存在、无虚假实体事件、**被搬记录自身**数据不丢。四个 bug 全部存在时这三条断言依旧全真——损坏面全部落在「被搬记录**携带**什么」与「**世界的其余部分**怎样了」上：

- seed 270：被搬子树成员的 isolated n:n link 行被连坐物理删除——没有断言说"这条无关的 link 还在"；
- seed 424/446：显式 null 被 defaultValue 改写——没有断言说"没动过的列保持原值"；
- seed 114：merged link 被重新发号——没有断言说"link 的逻辑 id 跨物理操作稳定"；
- seed 187：同住者的配对 link 被清行销毁——同住者根本不在夹具里。

这就是 AI 规划文献里的**框架问题**在测试上的投影：手写测试写得出"效果公理"（操作改变了什么），写不出也不会去写"框架公理"（其余一切不变）。**"id 不会变"、"无关记录不受影响"这类用户本体论的公理，恰恰是从来没有人断言过的性质**——它们只能由闭世界预言机（快照 diff ⟺ 事件流、跨步 id 追踪）机器化地持有。fuzzer 抓到全部四个子家族，靠的正是预言机对**整个世界**做对账，而不是对操作焦点做断言。

### 2.3 契约的双侧登记缺口

默认值改写还有一层：**查询快照对 NULL 列省略键 + 写路径对缺席键应用默认值**，两个各自合理的约定在「快照回写」组合下变成静默改写。r21 F-1 已把「键缺席 ⟺ 库里 NULL」登记为快照完备性契约——但只登记了**读侧**消费方（本地谓词求值）；本轮暴露的是**写侧**消费同一形状时的同一歧义。「形状 ⟺ 语义」类契约登记时必须读写两侧的消费方同时枚举。

## 三、F-3 深水区：状态历史深度、探针辖区、预言机缺口

### 3.1 三步状态装配——手写测试到不了的深度

查询面幻影（`b.out4` 返回从未 link 的记录）需要先装配出「同住未配对」行，其最短路径是**三步有状态历史**：依赖侧领养（B 进 D 的行）→ hub 亡故（D 删除，B/C 成为孤儿同住）→ 再配对（C 挂新 D）。手写测试的状态深度几乎恒为 1–2 步（创建→操作→断言）；**能到达损坏前置状态的历史长度是一根从未登记的轴**。fuzzer 的操作序列长度（30→40→50）就是这根轴的机器化——H 家族要 50 步序列才稳定现身，正说明状态深度每加一层，可达状态空间就多一圈人没见过的形状。

### 3.2 探针辖区 ≠ 结论辖区

r27b 的 F-5 守卫注释写着「sameTableReliance 会随行搬运，**L 探针验证**」。L 探针验证的是**单配对、目的行无同型碰撞**的形态；r28 的 seed 136/369 证明结论在「目的行已有同型记录」「子树成员自带其他配对」下不成立。**探针的辖区就是结论的辖区**——把探针结论写进守卫注释时，必须同时写明探针覆盖的形态边界，越界的格要么补探针要么按未验证处理。这是 r27 规则 7（方言匹配探针）的推广：不止方言，一切「已验证」声明都要标注验证域。

### 3.3 fuzzer 自己的预言机缺口（本轮如实记账）

seed 369 是靠**删除面**（幻影级联 → 事件完备性红）间接暴露的；**纯读取面**的幻影（不删除、只查询）fuzzer 至今判不了——结构不变量里没有「嵌套读取 ⟺ findRelationByName 一致」这条规则。该缺口本轮由人工家族分析 + 定向探针补上（`query_phantom_pairing` 形态已固化进 r28 回归），但机制化的收口应是**预言机第 8 条：配对读取一致性**（对每条 combined 关系断言两个读取面的答案一致）。квality-plan §2.2 的「行搬迁 count 对账」是它的兄弟条目，一并排入。

## 四、F-4 与 I-1/I-2：登记册回填的量词错误 + 契约的拓扑全积

- **F-4/I-2（扇出双跑）**：r27 把「单事件 × 多监听扇出」轴登记为「property dep 合并 / records dep 独立」——**按当轮咬人的实例展开取值，而不是按规则的全称量词展开**。规则本身（"每 X 注册一个 Y ⇒ 必须回答多 Y 同时命中执行几次"）是完备的，量词覆盖 eventDeps 轨与同源 records dep 退化格，但登记时没有机械展开。值得注意的是这两格**不是 fuzzer 抓的**（fuzzer 不生成计算声明），是**主动轴审计**（r27 复盘 §四.1「走查实现分支点」的执行）抓的——r27 提出的两个机制（生成器 + 轴审计）本轮各自交付了自己辖区内的致命项，相互不可替代。
- **I-1（reliance 拓扑等价）**：r27b 的 reliance 契约回归只铺在 combined 拓扑上（F-4/F-5 在那里被发现）。契约（依赖随 owner 死亡、re-parent 合法、displacement 致命）是**拓扑无关的语义**，回归矩阵却是拓扑特定的。r27 F-1 的教训（一种编译下的绿灯对另一种编译零置信度）在「载荷合法性」层学会了，没有迁移到相邻的「关系生命周期语义」层。**同一根轴在不同语义层反复咬人 ⇒ 它不该按语义层逐个回填，而应升格为固定审计维度**：任何声明面语义在定义时即对全部物理编译求全积（本轮的 8 轨 × 3 拓扑矩阵是可复用样板）。

## 五、对防御体系本身的结论

27 轮建起来的三支柱（预言机 / 登记册 / 汇合点）+ r27 新增的两机制（生成器 / 主动轴审计）在本轮的实战成绩单：

| 侦测机制 | 本轮交付 | 失手处 |
|----------|---------|--------|
| fuzzer（生成域扩张：种子 8→499、操作 30→50） | 全部 storage 侧致命家族（G+H） | 不生成计算声明 ⇒ 运行时扇出失明；缺读取一致性预言机 ⇒ 纯查询面幻影失明 |
| 主动轴审计（实现分支点走查） | Transform eventDeps / Custom 同源 dep 双跑 | 依赖审计者对"每 X 注册一个 Y"模式的识别，非穷尽 |
| fix-the-class 清单（兄弟面 sweep） | clone(deep) 三兄弟、reliance 拓扑矩阵、host-attr 轨接线 | 清单由修复触发，不自主发现新家族 |
| 手写特性测试（2000+ 用例） | 零新发现；防倒退全绿 | 本文档通篇 |

结构性的结论：**手写特性测试的角色已经完成了历史性转变——从"发现 bug 的主力"变成"契约的防倒退固化物"**。发现新 bug 的产能已完全由三个机器化/清单化机制承担，而它们各有辖区且辖区不重叠。下一步扩张按各自的短板排：fuzzer 纳入计算声明与读取一致性预言机（quality-plan §1.3 既定路线）、轴审计固定为每轮前置步骤、探针结论标注辖区。

## 六、本轮已落地的机制改造清单

1. Setup 期行槽位排他不变量（`combineRecordTable` co-located 检测——检测点就在旧早退处，非法状态从源头不可制造；mergeLinks fail-fast + reliance 降级）；
2. 配对真相源统一到 link id 列，消费面全枚举收口（synthetic `&` + 读取剪枝 + match 守卫原子 + 删除核验补载 + `physicalRowMatch/physicalRowRead` 显式物理寻址豁免——**豁免必须显式**，flashOut 的物理寻址是特性不是漏网）；
3. `clearRowDataForMigration`（搬迁与删除的机制分离）+ NULL 物化 + 携带 link id 保号 + relocate 端点翻转；
4. 搬运子树级 co-tenant 守卫统一（host-attr 轨接线 + `collectOutOfSubtreeCombinedPairings` 子树谓词）；
5. reliance 槽位守卫收敛于 `createRecord` 入口 + 方向感知的 owner 解除（拓扑等价矩阵 8 轨 × 3 拓扑固化）；
6. 事件驱动计算 (computation, event) 去重（live + migration 两消费方）；
7. Custom 同源多 dep 声明期 fail-fast（planIncremental + depKey 显式分流面以回归固化）；
8. clone(deep) 家族对齐（BoolExpressionData/Conditions/Activity/ActivityGroup）；
9. 登记册三根新轴/扩展（扇出轴按量词补全取值、物理搬迁≠逻辑删除、同住≠配对）。

## 七、给后续轮次的操作性规则（增补）

1. **编码手段 ⇏ 事实**：任何「用物理布局编码逻辑事实」的设计（同行=配对、列非空=存在、行序=顺序……），必须显式登记事实的**唯一真相源列**，并枚举全部消费点验证它们读同一真相源；物理寻址消费点必须显式豁免。
2. **写守卫先找不变量**：守卫落地前问「这条守卫是哪个不变量的影子」，按不变量的键空间设防，而不是按咬人实例的键空间。
3. **复用前判语义前提**：复用带副作用的例程前，回答「两个调用方对副作用（级联/默认值/发号/事件）的语义前提是否同构」。
4. **框架公理进预言机**：用户本体论的公理（id 稳定、无关记录守恒、删除不复活）不许依赖手写断言——只能由闭世界预言机持有；新增预言机规则优先补公理类（下一条：配对读取一致性 + 行搬迁 count 对账）。
5. **状态历史深度是轴**：损坏前置状态的最短装配步数登记入册；手写回归对 ≥3 步装配的家族必须复用 fuzzer 种子或固化装配助手。
6. **探针辖区 = 结论辖区**：一切「已探针验证」的守卫注释必须标注探针覆盖的形态边界，越界按未验证处理。
7. **登记册回填按全称量词展开**：新轴入册时机械枚举规则的全部取值（含未咬人的），不按本轮实例。
8. **拓扑等价是固定审计维度**：每个声明面语义（合法性、生命周期契约、事件契约）在定义时即对全部物理编译 × 轨道求全积。
9. **生成域扩张即审查**：每轮至少做一次 fuzzer 生成域扩张（种子池 / 操作数 / schema 菜单三选一）并收口新红种子；固定 CI 种子集只防倒退，不产生新知识。
10. **组合契约双侧登记**：「形状 ⟺ 语义」类契约登记时，读侧与写侧消费方同时枚举。
