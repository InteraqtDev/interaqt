# 深度反思：为什么 17 轮审查、1850+ 用例的测试体系没有测出 r17 的四个致命 bug

- 日期：2026-07-11
- 关联：`deep-review-2026-07-11-r17.md`（问题发现与修复）
- 性质：测试体系的结构性复盘。不是「谁漏写了哪个用例」的归因，而是回答「**是什么样的体系性盲区让这些用例系统性地不会被写出来**」。

---

## 〇、先给结论

四个 bug 各自有一个直接的「缺失格子」，但把四个格子放在一起看，暴露的是四个**结构性盲区**，按重要性排序：

1. **物理拓扑是测试矩阵的隐藏维度**——框架把逻辑声明编译成三种不同的物理布局（combined / merged / isolated），bug 生活在物理层，测试写在逻辑层，fixtures 把拓扑焊死在单一取值上（F-1、F-2）。
2. **三个观察面中事件面没有预言机**——查询面有「增量==全量」预言机（r6）、引用完整性面有「无悬挂端点」预言机（r7），事件面的完备性契约（「每次持久化变更必有对应事件」）从未被结构性验证（F-2）。
3. **交叉格子按 bug 驱动补点，不按维度做积**——「对称关系」和「带 callback」两个维度各自有覆盖，交叉格为空；十七轮形成的模式是「发现 bug → 修 → 为那个点补回归」，邻域从不扫描（F-3）。
4. **内部假设以注释形式存在而非可执行断言**——「路径中只可能有一个对称关系」这句错误的断言以注释形式存在于代码中，从未被挑战；假设失效时的表现是静默半结果而不是受控错误（F-4）。

下面先逐个 bug 做「为什么恰好是它逃逸」的显微镜分析，再展开四个结构性盲区，最后给出已在本轮落地和建议后续落地的改进。

---

## 一、逐 bug 显微镜：每个 bug 恰好穿过了哪些防线

### F-1（x:1 merged-link 抢夺不解除旧 owner）穿过了三道防线

**防线一：oneToOne.spec.ts 的拓扑被 fixture 焊死。**
该文件覆盖了 1:1 关系的 create/update/delete/抢夺（"f2 应该仍然只有一个，是被移动了的"这条断言明确测了抢夺语义）——但它的 setup 是：

```ts
setup = new DBSetup(entities, relations, db, ['Profile.owner'])   // 显式 mergeLinks
```

显式 `mergeLinks` 产生 **combined（三表合一）拓扑**，抢夺由 flashOut 处理（这条路径是健康的）。而**不传 mergeLinks 的默认拓扑**（`Setup.mergeRecords` 第 3 步：非 reliance x:1 关系表合并进宿主行，FK 列在宿主行上）走的是完全不同的代码路径——`addLink` 的自动 unlink 条件 `!isCombined() && !isMerged() && (isManyToOne || isOneToMany)` 把 merged 与 1:1 双双排除。**同一个逻辑断言，在默认拓扑下从未被执行过。**

**防线二：r6/r7 矩阵的「抢夺」格子没有拓扑维度。**
r6-F3 修复了 isolated 1:n 抢夺崩溃、r7 补了对称关系的双侧删除——每次都是在**单一拓扑**上修复并回归。「抢夺」这个操作维度从未与「物理拓扑」维度做过笛卡尔积扫描。

**防线三：referentialIntegrityMatrix 的不变式差一步就能抓住它。**
r7 引入的 INV-1（无悬挂端点）/ INV-2（对称双向可见）是「不信任写路径」的正确设计，但它缺一条 **INV-3：x:1 关系的排他侧唯一**（每个 target 至多一个 owner）。而且矩阵的 1:1 格子只放了 self-referential reliance 退化点，没放「引用已被占用目标」的抢夺退化点。**预言机的形状对了，不变式清单和格子清单不全。**

### F-2（同 id + `&` 原地更新零事件）穿过了两道防线

**防线一：事件面没有普适预言机，只有正向点断言。**
现有测试断言事件的地方，都是「已知此处应有事件」的正向格子（create 应有 create 事件、replace 应有 delete+create）。「**不该沉默的地方沉默了**」是否定形命题，逐点断言永远列不完。唯一能抓住它的是结构性的**事件完备性预言机**：DB 前后快照的 diff 必须被事件流完全解释。这个预言机从未存在——于是「数据面写了、事件面没发」这一整类 bug（本质上是写路径与事件路径的条件不一致：`getSameRowFieldAndValue` 无条件写列，`preprocessSameRowData` 按「id 是否变化」发事件）没有任何防线。

**防线二：同一语义的两种写法没有等价性测试。**
「更新关系属性」有两种合法表达：`updateRelationByName(...)`（有测试，传播正常）和「经宿主实体 + 同 id + `&`」（只测过 create 时携带 `&` 和 replace 时携带 `&`——r5-F-3/r6 修的正是后者，**same-id in-place 这第三形态从未被测**）。r6 矩阵有「update 终态 == create 终态」的对偶预言机，但它对比的是**查询结果终态**，不是**事件流等价**——数据面确实写对了，所以对偶预言机是绿的。

另外值得诚实记录：本轮修复时还发现该形态在 **combined 拓扑下连数据面都是坏的**（flashOut 把旧行数据 merge 回来覆盖新值，`title: 'new'` 静默变回 `'old'`）——同样零测试。这再次印证拓扑是隐藏维度：同一写法在 merged 拓扑下「数据对、事件缺」，在 combined 拓扑下「数据错、事件缺」，两个格子都是空白。

### F-3（对称关系 + callback 聚合状态碰撞）穿过了两道防线

**防线一：两个维度各有覆盖，交叉格为空。**
- `symmetricRelation.spec.ts`：对称关系 × Count **无 callback**（`Count.create({record: friendRelation})`）——通过，因为无 callback 时没有逐项状态（`itemStateKey` 为 null），存在性 delta 恰好绕开碰撞；
- `count-callback.spec.ts` 等：callback 聚合 × **非对称关系**——通过，因为一条 link 行只服务一个宿主。

「对称 × callback」交叉格触发的条件是**实现内部的分支**（有 callback → 启用以 link 行为 key 的 `RecordBoundState`）。黑盒视角下「加个 callback」看起来只是换一种计数方式；白盒视角下它切换到一条**引入新 key 空间假设**（一行一宿主）的路径。**测试矩阵的维度取值没有随实现内部状态机制的引入而更新**——`RecordBoundState(relation.name)` 落地时，没有人回头问：「以关系行为 key 的状态，在哪些关系形态下一行会对应多个宿主？」答案（对称关系）在 r7 已经作为「关系维度的退化点」被总结过。

**防线二：r7 的教训被记录、未被制度化。**
r7 报告原话：「退化点（self / empty / boundary）应是每个维度的默认必选值」。这个教训应用到了当轮的写路径（删除/update 的对称双侧匹配），但没有变成任何机制——r6 聚合一致性矩阵的关系维度取值仍是 {n:n, 1:1, 1:n}，没有补 {对称 n:n}；callback 有无也从未成为维度。**复盘产生的知识以文字形式留在报告里，没有回灌进矩阵生成器的维度清单。**

### F-4（双对称段路径只展开第一段）穿过了两道防线

**防线一：错误假设以注释存在，从未被断言化或挑战。**
`MatchExp.buildFieldMatchExpression` 中的注释：「路径中只可能有一个 n:n symmetric 关系。因为路径中有多个的在语义逻辑上就不正确」——这是一个**错误的语义判断**（`friends.friends`「朋友的朋友」是社交域最自然的两跳查询），以注释形式存在多年。如果当初把它写成 fail-fast 断言（「检测到多段对称路径 → 抛明确错误」），F-4 会在第一个用户查询时以受控错误暴露，而不是静默返回半结果。**explicit control 原则被严格应用于用户输入，却没有应用于内部不变量。**

**防线二：「增量==全量」预言机在这里结构性失明。**
r6 预言机对比「增量维护的聚合值」与「从 storage 全量重算的值」。F-4 的错误发生在**脏宿主定位**（Scheduler 用 `friends.friends.&.id` 查询找该重算谁）：没被找到的宿主**根本不会触发重算**，增量值与全量值都停留在旧值——预言机比较的两个数**一起错**，绿灯。r7 已经认识到该预言机对「写路径自洽腐蚀」失明（这是引用完整性矩阵的由来），但**查询编译器层**的同型失明（增量与全量共享同一个错误的查询结果）没有对应的差分防线（如「SQL 查询结果 == 内存朴素过滤结果」的对照）。

## 二、四个结构性盲区（超越单个 bug 的模式）

### 盲区 1：物理拓扑是隐藏维度——测试的抽象层级与 bug 的抽象层级不匹配

interaqt 的核心特性之一是把逻辑声明（Entity/Relation）编译成优化的物理布局：`Setup.mergeRecords` 自动决定三表合一（combined）、关系表并入端点行（merged to source/target）、或独立关系表（isolated）。**这意味着每一个写路径行为都有至少三个物理实现**，分别走 flashOut / FK 列写入 / link 行增删三条代码路径。

但测试套件几乎全部写在逻辑层：fixture 声明实体关系，断言逻辑行为。拓扑由 fixture 的 `mergeLinks` 参数（或默认策略）**焊死在单一取值**上，且这个选择对读测试的人几乎不可见（`['Profile.owner']` 一个字符串参数决定了整个文件测试的是哪条代码路径）。十七轮里 merged 拓扑的写路径 bug 反复出现（r5-F-3、r6 家族、本轮 F-1/F-2），根本原因一直是同一个：**逻辑层的测试矩阵天然无法覆盖物理层的分支**。

类比：这相当于测试一个编译器时，所有测试都只跑 `-O0`。

### 盲区 2：三个观察面，只有两个有预言机

同一次变更有三个可观察面，每个面服务不同的下游：

| 观察面 | 下游 | 预言机 | 状态 |
|--------|------|--------|------|
| 查询面（find 结果） | 应用读取 | r6「增量==全量」+ 常规断言 | ✅ 有 |
| 引用完整性面（link 端点） | 数据不变量 | r7 INV-1/INV-2 | ✅ 有（不变式清单不全） |
| **事件面（mutation events）** | **全部响应式计算** | 无 | ❌ **零结构性防线** |

事件面恰恰是这个框架**最** mission-critical 的面——响应式是框架的存在理由，而事件是响应式的唯一输入。「每一次持久化变更必有对应事件、每个事件对应真实变更」这条契约的破坏（F-2）意味着计算静默陈旧，是所有错误形态里对用户伤害最大的（无报错、查询正确、计算错误）。这个面没有预言机不是偶然疏忽：事件完备性是「否定形」命题（不该沉默处不沉默），逐点正向断言的测试范式天然写不出来，必须用「快照 diff ⟺ 事件流」的结构性对账。

### 盲区 3：点状补测 vs 邻域扫描——修复模式本身在制造盲区

十七轮形成的稳定工作流：深度审查发现 bug → 编写该 bug 的最小复现 → 修复 → 复现翻转为回归测试。这个流程对**已发现**的 bug 是健康的，但它有一个隐性副作用：**每轮的测试增量都是「点」，而 bug 的分布是「维度组合空间中的洞」**。r6-F3 修 isolated 1:n 抢夺时，merged 1:1 抢夺（同一操作维度、相邻拓扑取值）就在旁边，没有被扫描；r5-F-3 修 replace 场景丢 `&` 时，same-id 场景（同一载荷维度、相邻操作取值）就在旁边，没有被扫描。

r6/r7 的矩阵是对这个模式的正确反叛（一次把一个维度积扫完），但矩阵建成后就静止了——新发现的维度（对称性、callback 有无、拓扑）没有作为新的取值/新的轴回灌进已有矩阵。**矩阵需要的不是一次性建设，而是「每轮审查产出的新维度必须回灌」的持续义务。**

### 盲区 4：内部假设的口头化——注释里的「只可能/不可能」没有执行力

本轮两处直接相关：`MatchExp` 的「路径中只可能有一个对称关系」（错误假设 → F-4 静默半结果）、`addLink` 的 unlink 条件把 `isMerged()` 排除（隐含假设「merged link 的排他性总由别处处理」——对 FK 侧成立、对 1:1 非 FK 侧不成立 → F-1）。两个假设如果在写下时就断言化（不支持就 fail-fast），都会把静默错误转为声明期/首次使用期的受控错误。框架对**用户**的假设校验非常严格（守卫链 boolean 契约、payload 校验、fail-fast 家族），对**自己**的假设却停留在注释——这是 explicit control 原则的应用不对称。

## 三、本轮已落地的改进

### 3.1 随修复落地（点与邻域）

1. **修复即扫邻域**（针对盲区 3）：`review-repro-r17.spec.ts` 的修复回归不止覆盖四个 bug 点，同步固化了兄弟格——combined 拓扑的 same-id 嵌套更新（数据面+事件面）、combined 抢夺对照组、merged 拓扑从 attribute 侧抢夺、replace 对照组（delete+create 事件而非 update）、幂等重写对照组（零事件且 link 数据保留）、n:1 非排他对照组（**不**误删多源）。
2. **假设断言化/能力化**（针对盲区 4）：多段对称路径从「注释宣称不可能」改为**真实支持**（笛卡尔展开，含防二次展开守卫）；对称+逐项状态从「静默碰撞」改为显式守卫退回全量重算（正确性优先，性能代价限定在对称×callback 格子内）。

### 3.3 结构层测试的即时收益（矩阵首跑抓出的新 bug）

结构层不是「为已修 bug 补票」——首跑就抓出三个**此前十七轮未发现**的既有 bug（均已修复）：

| 新 bug | 发现者 | 修复 |
|--------|--------|------|
| 对称 fan-out 下 `&` 数据错挂：对端有多条边时，父子边的 `&` 可能挂成对端另一条边的数据（C 查 friends 拿到 A—B 边的 weight） | 对称聚合矩阵的朴素预言机 | `QueryExecutor.findXToManyRelatedRecords` 按「两端点 id 与父子匹配」选取方向变体（AttributeQuery 对对称 link 强制附带端点 id 作判据） |
| combined 拓扑删除端点实体留**悬挂 link**：link 列不随行清除，delete 事件已发但 `findRelationByName` 仍可见 | 事件完备性预言机（写路径拓扑矩阵首跑） | `DeletionExecutor` 同行删除时一并清除 combined link 的管辖列 |
| combined link 经 addRelation 抢夺时崩溃（OR 匹配把无关行也 flashOut）+ 旧业务 link 的 delete 事件漏发（记在虚拟 link 名下） | 写路径拓扑矩阵 | `flashOutCombinedRecordsAndMergedLinks` 按 ref id 校验行归属 + 补发业务 link delete 事件 |

**变异测试证明检出力**：把 src 整体回退到修复前基线（`b75c21ec`）、保留新矩阵运行——
16 个矩阵用例中 **9 个变红**，精确命中 F-1（merged 双 owner、addRelation 双 owner）、
F-2（combined 事件缺失）、F-3（对称计数 0）、F-3×F-4（2 跳收敛）、F-4（2/3 跳半结果）
全部四个家族；变绿的 7 个是修复后语义（1:n/n:1/n:n 对照组）本就健康的格子。
即：**若这套矩阵在 r16 就存在，r17 的全部四个致命 bug 会在提交前被自动发现。**

## 五、一句话总结

**这些 bug 不是「测试写少了」，而是测试体系的坐标系里缺了几根轴：物理拓扑轴、事件完备性轴、以及「维度取值必须包含退化点并随实现演化更新」的维护义务。** 十七轮审查修复的绝大多数问题最终都能归到同一句话——合法声明组合的空白格；要终结这个模式，需要的不是更多的点状测试，而是把「格子从哪里来」这件事本身制度化。
