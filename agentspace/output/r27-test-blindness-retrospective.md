# 深度反思：为什么已有的测试系统没有检测出 r27 的这些问题

- 日期：2026-07-13
- 关联：`deep-review-2026-07-13-r27.md`
- 性质：测试与修复体系的结构性复盘 + 机制改造记录。覆盖本轮全部七个问题（F-1 / F-2 / I-1–I-5），重点解剖两个致命家族。

---

## 〇、先给结论

r25/r26 的致命项形状是「同一契约的兄弟消费方/兄弟分支漏网」——契约已知，实例没铺满。
本轮两个致命项的形状**更深一层**：

**轴本身从未被登记。不是格子漏铺，是维度表里没有这根轴。**

| 类 | 结构性失明 | 本轮案例 |
|----|-----------|---------|
| A | **维度缺失**：矩阵有拓扑轴、有载荷轴，但两轴只在深度 1 交叉；「子记录载荷自身携带什么」从未成为取值 | F-1 |
| B | **集合性质失明**：所有监听面断言都是单监听性质（可达/合法），「同一事件命中同一计算的多个监听时执行几次」是监听**集合**的性质 | F-2 |
| C | **声明面与消费面的合法性分裂**：声明期确认合法，没有问「每个消费面是否接受」 | I-1、I-5 |
| D | **「声称已修」与「已接线」之间没有机器验证** | I-3 |

## 一、F-1 的显微镜：拓扑矩阵为什么给深嵌套发了假签证

### 1.1 事实：矩阵铺满了深度 1，深度 2+ 从未进过任何测试

`writePathTopologyMatrix.spec.ts`（r17 落地）把 1:1 的 combined/merged 两种拓扑 × 宿主直接属性的
全部写法（嵌套新建 / ref / 同 id `&` / null / 抢夺 / 删除）铺成矩阵，四个观察面齐全。
但每个格子的载荷都长这样：

```ts
{ name: 'u1', profile: { title: 'p1', '&': { viewed: 1 } } }   // 子记录只有 value 属性
```

「profile 里面还能放什么」——`teams: [{id}]`（isolated 关系）、`company: {id}`（merged-FK）、
`avatar: {url}`（更深的 combined）——**从来不是矩阵的一根轴**。

### 1.2 为什么没人怀疑：merged 拓扑的正确性提供了假签证

同样的深嵌套载荷在 merged 拓扑下**完整工作**（子记录经 `createRecordDependency → createRecord`
递归，自己的关系由自己的 create 处理）。文档（USAGE_GUIDE 的深嵌套示例）、大量既有测试
（各 spec 的嵌套 create）都在 merged/isolated 拓扑上跑——绿灯让「嵌套载荷已覆盖」在心理
账本上盖了章。

但 **拓扑轴的全部意义就是「同一逻辑声明在不同物理编译下必须等价」**。combined 子记录刻意
不走 createRecord（同行写入），于是 merged 靠递归天然获得的能力，combined 需要显式实现——
而实现只做了 value 列（`getSameRowFieldAndValue` 递归），其余结构没有任何执行者。
「表面可用」（值列写进去了、嵌套查询能读到）让手工冒烟测试也不报警。

### 1.3 runtime 可达性被低估

`mergeLinks` 是 storage 层参数，容易被当成「storage 内部配置的角落」；但 1:1
`isTargetReliance`（框架推荐的生命周期依赖声明）在 Setup 里**自动**三表合一——combined
拓扑是 runtime 用户的常规可达面。六种损坏形态全部在 PGLite + Controller 下复现。

### 1.4 机制回应

1. 登记册新增「**载荷嵌套深度 × 子记录拓扑**」维度（深度 ≥ 2 的全部取值 × merged/combined）；
2. 修复选择 fail-fast 而不是静默容忍或仓促全量实现：六种损坏形态全部转化为带 workaround
   的声明错误，工作面逐一以回归固化；
3. 拓扑等价性升格为操作性规则：**任何「同一声明、多种物理编译」的特性，测试必须按
   载荷形态 × 编译形态求全积**，一种编译下的绿灯对另一种编译零置信度。

## 二、F-2 的显微镜：监听集合的性质没有断言者

### 2.1 事实链

- `convertDataDepToERMutationEventsSourceMap`：每个 dataDep 注册独立监听——单独看合理；
- `buildComputationMutationListener`：逐 source 执行——单独看合理；
- 组合结果：一次 update 命中同一计算的两个 property dep → 同一事件对象执行两次 →
  `useLastValue` 增量 +2、create 初始计算双跑。

r18 的死监听不变量（recordName 可达）、r22 的 type 白名单——都是**单监听**的性质，
`assertListenerReachable` 逐监听断言。**「命中集合的执行次数」是监听集合的性质**，
没有任何断言者站在集合视角。

### 2.2 夹具偏置（r25 F-1 的「测试总带显式 `&`」同族）

既有多 dataDep 测试的每次 update 都只改**一个** dep 的字段——单 dep 命中格铺满了，
多 dep 同时命中格为空。写测试的人自然按「验证每个 dep 都能触发」组织用例，
「两个 dep 同时触发会怎样」不在直觉清单上。

### 2.3 用户层不可修复性是定谳关键

两次调用收到**同一个事件对象**（recordName/keys/record 全同）；`planIncremental` 的
`context.depKey` 不同，但增量结果已在第一次调用落库。判定「框架必须合并」的依据：
**用户层不存在任何可区分信息实现幂等**。反面（records dep 不去重）同样有明确依据：
不同 dep 的 match 使 per-dep membership 判定（entered/left）是增量语义的组成部分。

### 2.4 机制回应

1. 登记册新增「**单事件 × 多监听扇出**」维度（property dep 合并 / records dep 独立 /
   异 targetPath 独立；live Scheduler 与 migration 链式 rebuild 两个消费方）；
2. 修复落在两个消费方的同一语义（Scheduler 去重 + migration 入队/重放双层去重）；
3. 回归同时铺「该合并的合并」与「不该合并的没被合并」两个方向（r21 复盘的 skip 双向规则
   在扇出维度上的应用）。

## 三、I 组的公共形状

- **I-1 / I-5（声明面与消费面分裂）**：r26 声明期确认单边包装合法，但没有枚举「谁读这个
  声明」——`evaluate`/`evaluateAsync`/`map`/`SQLBuilder.buildWhereClause` 四个读者全部抛错/崩溃。
  onlyRelationData 有类型位、有合并语义（r17 R-1 还修过合并丢标志）、有存在性测试——
  唯独没有测试断言**声明了它之后返回什么**。「字段被认真对待过」≠「字段有实现」。
  尤其危险的放大器：`.cursor/rules` 把该位档成 "is collection"——**面向 agent 的文档错误
  会被系统性执行**，比面向人的文档错误传播快得多。
- **I-2（运行期静默偏好）**：`property` 在场时 `record` 是死配置——「二选一」在 create 里
  只查了「至少一个」。矛盾声明的拒绝必须与「缺失声明的拒绝」同时写，这是 XOR 的两半。
- **I-3（声称已修 ≠ 已接线）**：r26 写了 helper、写了注释、写了报告——没有写**该修复独有
  行为的运行时探针**（MySQL 面的 operationKey 测试为零）。r24 复盘说「从未运行的测试面 =
  从未验证的声明」；本轮的变体是「**从未被调用的修复 = 从未存在的修复**」。机制回应：
  修复代码若声称改变某驱动/某路径的行为，回归必须在该驱动/该路径上跑出与修复前不同的结果
  （本轮的 MySQL operationKey 回归在真实 MySQL 8 上验证 400+ 字符键 round-trip 与前缀不碰撞）。

## 四、本轮已落地的机制改造清单

1. 登记册两根新轴：「载荷嵌套深度 × 子记录拓扑」「单事件 × 多监听扇出」（`WritingComputationTests.md`）；
2. `preprocessSameRowData` 汇合点守卫：combined 子记录的不支持结构全部 fail-fast（六种损坏形态 → 声明错误）；
3. Scheduler + migration 双消费方的扇出去重（同一契约两处收口）；
4. BoolExp 单边 and/or 的四个读者统一左透传语义；
5. `validateAggregationTarget` 汇合点（六聚合类 XOR）；
6. MySQL operationKey 代理键真实接线 + env-gated 运行时探针进 CI；
7. 文档止损：RealTime 调度承诺如实化、storage-layer.mdc 第三元组勘误。

## 五、给后续轮次的操作性规则（增补）

1. **「同一声明、多种物理编译」的特性，按载荷形态 × 编译形态求全积**——一种编译下的绿灯
   对另一种编译零置信度（F-1）。
2. **凡是「每 X 注册一个 Y」的机制，必须显式回答「多个 Y 同时命中时执行几次」**——
   集合语义不会从单元素断言中涌现（F-2）。
3. **声明期放行任何形态前，枚举该形态的全部求值/编译读者并逐一验证**（I-1）；
   公开类型面上的每个字段必须有「声明它之后行为如何变化」的断言，存在性测试不算（I-5）。
4. **修复声明必须伴随该修复独有行为的运行时探针**，探针跑在修复声称生效的驱动/路径上（I-3）。
5. **面向 agent 的规则文档（.cursor/rules 等）与实现的一致性是测试面**——错误会被系统性执行（I-5）。
