# 深度反思：为什么已有的测试体系没有检测出 r27 的这些致命 bug

- 日期：2026-07-13（同日深化修订）
- 关联：`deep-review-2026-07-13-r27.md`
- 性质：测试与修复体系的结构性复盘 + 机制改造记录。覆盖本轮全部七个问题（F-1 / F-2 / I-1–I-5），重点解剖两个致命家族，并回答一个更根本的问题：**为什么 27 轮审查过去了，测试体系仍然按轮产出致命 bug**。

---

## 〇、先给结论：本体论错位（ontology mismatch）

27 轮复盘的教训各有名字（驱动差异轴、第二消费轨、兄弟分支、假签证……），但它们是同一个根因的不同投影：

> **测试是用「声明本体论」写的，bug 住在「编译本体论」里。**

interaqt 的核心承诺是声明与物理编译分离：用户声明数据"是什么"（Entity/Relation/payload/dataDeps），框架决定"怎么实现"（物理拓扑合表、监听注册、方言分支、增量/全量路径）。写测试的人在**用户可见的本体论**里思考——"嵌套创建测过了""多依赖测过了""迁移键测过了"；而实现沿着**用户不可见的本体论**分裂等价类——同一个嵌套载荷在 merged 拓扑走 `createRecord` 递归、在 combined 拓扑走同行写入；同一个"多依赖"声明按 dep 数量注册 N 个监听；同一个 operationKey 在 PGLite 直存、在 MySQL 走哈希分支。

**一条声明面的绿灯，只覆盖它实际走到的那一个编译等价类，却在心理账本上给整个声明面盖章。** 抽象能力越强（一个声明面下藏的物理变体越多），单个绿灯辐射的假置信度越大。这不是测试写少了的问题——这是本框架的核心设计原则在系统性地**制造**盲区，而测试组织方式（按用户特性组织）恰好看不见这种制造。

r25/r26 与 r27 的递进关系值得精确命名：

| 轮次 | 失明形态 | 已知度 |
|------|---------|--------|
| r25/r26 | **缺格**：轴已登记（create payload 契约、delete 端点契约），某个消费方/兄弟分支的格子没铺 | 契约已知，实例漏网 |
| **r27** | **缺轴**：「载荷嵌套深度」「监听扇出语义」这两根轴从未进过维度登记册——不是格子空着，是维度表里没有这一列 | 契约本身不存在 |

缺格可以靠"fix the class 清单 + 家族 sweep"补；**缺轴补不了——你无法枚举你没想到的维度**。这把问题推向下一个层次：轴从哪里来？答案在第四节。

## 一、F-1 的显微镜：五道防线如何各自失守

### 1.1 事实基础

`writePathTopologyMatrix.spec.ts`（r17 落地）把 1:1 的 combined/merged 拓扑 × 宿主直接属性的全部写法铺成矩阵，四个观察面（查询/事件/计算/不变量）齐全。但每个格子的载荷都长这样：

```ts
{ name: 'u1', profile: { title: 'p1', '&': { viewed: 1 } } }   // 子记录只有 value 属性
```

「profile 里面还能放什么」——`teams: [{id}]`（isolated）、`company: {id}`（merged-FK）、`avatar: {url}`（更深的 combined）——从来不是矩阵的取值。

### 1.2 防线逐一检视

1. **维度登记册：「载荷形态」轴的定义本身是扁平的，而载荷是递归结构。** 登记册里"nested 新建"是一个取值，但它没有回答"nested 的东西自己还能带什么"。**递归结构的维度被当成了标量**——这不是漏填格子，是轴的类型签名写错了。深度是载荷形态的隐藏参数。

2. **merged 拓扑的绿灯发放了跨拓扑假签证。** USAGE_GUIDE 的深嵌套示例、大量既有测试的嵌套 create 全部跑在默认（merged/isolated）拓扑上——那里 `createRecordDependency → createRecord` 递归让深嵌套天然工作。"嵌套创建已覆盖"于是在心理账本上盖了章。但**拓扑轴的全部语义就是「同一逻辑声明在不同物理编译下必须等价」**——这要求载荷形态 × 拓扑求全积，一种编译下的绿灯对另一种编译的置信度严格为零。这正是〇节本体论错位的教科书实例。

3. **部分成功比彻底失败更难被发现。** combined 子记录的 value 列经 `getSameRowFieldAndValue` 递归**确实写进去了**——嵌套查询能读到数据，冒烟测试、手工验证全部通过。损坏面只暴露在：link 记录身份（按名 find）、事件流（需要预言机）、聚合可见性（需要响应式断言）。bug 的可发现性 = 输入到达 × 损坏面被观察 × 断言足够强，三个因子的乘积；矩阵格子内三因子齐全，格子外的 ad-hoc 测试通常只看查询面——恰好是唯一"表面可用"的面。

4. **预言机只验证发生了的执行；生成执行是矩阵的职责——缺口在生成侧。** 事件完备性预言机（r17 起）足够强：任何人把「combined 子记录带关系」的写操作放进 `withEventCompleteness`，缺失的 link 事件当场变红。它没响是因为**这个输入形状从未被生成过**。r17–r26 把预言机侧建得很强（第 5/6/7 条规则、朴素重算对照、不变量断言），生成侧仍然是人肉枚举——人枚举不出自己没想到的形状。

5. **可达性误判压低了怀疑等级。** combined 拓扑在心理上被归档为"storage 层 mergeLinks 配置的角落"；实际上 1:1 `isTargetReliance`（框架推荐的生命周期依赖声明）在 Setup 里**自动**三表合一——runtime 用户的常规可达面。六种损坏形态全部在 PGLite + Controller 下复现。

6. **缺一条守恒律不变量。** `NewRecordData` 递归分类了**完整**的载荷树（数据结构是全的），executor 只消费宿主层一级——分类树上静静挂着永远无人消费的死枝。「**凡被分类的必须被消费或被拒绝**」是一条可检查的守恒律，此前只存在于"实现恰好如此"的默契中。本轮的 fail-fast 守卫就是这条守恒律的追认。一般化：**任何"递归构建、迭代消费"的结构对，其差集就是 bug 蓄水池**——值得对 codebase 做一次同构扫描。

## 二、F-2 的显微镜：集合性质没有断言者 + 夹具的幂等性遮蔽

### 2.1 单监听性质 vs 监听集合性质

r18 的死监听不变量（recordName 可达）、r22 的 type 白名单——全部是**单个监听**的性质，`assertListenerReachable` 逐监听断言。「同一事件命中同一计算的多个监听时执行几次」是**监听集合**的性质：每个组成部分单独看都合理（每 dep 注册监听：合理；逐 source 执行：合理），bug 从组合中涌现。逐元素断言对涌现性质天然失明——这类性质必须显式写成系统级契约（"一次逻辑变更 = 一次计算执行"），而这条契约此前不存在于任何文档、注释或测试。

### 2.2 夹具形态的双重遮蔽（本轮证据最扎实的部分）

检视全部既有 Custom 测试（`custom.spec.ts`、`schedulerEdgeCases.spec.ts` 等）发现两层系统性偏置：

1. **声明形状偏置**：既有夹具全部用**单个 property dep 携带多字段 attributeQuery**（`_current: { attributeQuery: ['basePrice', 'taxRate', 'discount'] }`）——注册一个监听，永不双跑。「N 个独立 dep」的声明形状（API 完全允许，`incrementalDataDeps` 的文档注释还专门讨论过多依赖）从未在任何测试中以"两个同宿主 property dep"的形态实例化。
2. **观察函数偏置**：既有夹具的 compute/incrementalCompute 几乎全部是 **dataDeps 的纯函数**（重算式）——对重复执行**幂等**。就算哪个夹具偶然踩中双跑，值断言照样绿。只有 `useLastValue` 的累加器或执行计数 spy 能暴露执行次数；而恰好没有任何夹具同时具备"多 dep + 非幂等"两个特征。

这与 r25 F-1 的「测试总带显式 `&`」、r26 F-1 的「夹具偏置到已正确的兄弟路径」同族，但机制上更隐蔽：**夹具的函数式纯度本身成为了遮蔽层**。执行次数是反应式框架的一等语义（增量语义 = 恰好一次），而纯夹具把它投影没了。

### 2.3 定谳依据：用户层不可修复性

两次调用收到**同一个事件对象**（recordName/keys/record 全同）。判定"框架必须合并"的依据不是审美，而是：用户层不存在任何可区分信息实现幂等。反面（records dep 不去重）同样有明确依据：不同 dep 携带不同 match，per-dep 的 membership 判定（entered/left/skip）是各自增量语义的组成部分。语义按 dep 类型分裂，两个消费方（live Scheduler / migration 链式 rebuild）同一契约收口。

## 三、I 组的公共形状（每个都指向一条流程缺口）

- **I-1（合法化决策没有触发读者枚举）**：fix-the-class 清单的第 1 条要求"枚举声明面的全部读者"——但清单只在**修 bug 时**被执行。r26 I-3 做的是一个**合法化决策**（"单边包装是既有合法写法，不在此收紧"），决策改变了契约面，却没有触发"谁在求值期读这个形状"的枚举——四个读者（evaluate/evaluateAsync/map/SQLBuilder）全部会炸。**契约面的任何决策（合法化、收紧、废弃）都应触发与修复同等的读者枚举**，而不只是修复。
- **I-5（存在性测试 ≠ 行为断言）**：`onlyRelationData` 有类型位、有合并语义（r17 R-1 还修过合并丢标志！）、有测试（`queryExecutorEdgeCases` 的用例）——唯独没有任何断言说**声明了它之后结果里有什么**。那个测试只断言 `found.id`，名字还叫 "skips entity query"——把 bug 的行为当成了特性名。公开类型面上的每个字段必须有"声明它 vs 不声明它，行为差异是什么"的断言；被认真对待过（review 过、合并逻辑修过）不等于有实现。放大器：`.cursor/rules` 把该位错档成 "is collection"——**面向 agent 的规则文档会被系统性执行**，其错误传播速度远超面向人的文档。
- **I-3（假绿：测试跑在兄弟分支上）**：这是本轮最值得警惕的失守形态。r26 的 operationKey 修复**有测试且测试是绿的**——`review-fixes-2026-07-13-r26-leftovers.spec.ts` 的 L-1 用例跑在 **PGLiteDB** 上，而 `migrationOperationKeyForStorage` 对非 MySQL **按设计**原样返回——被测的恰好是不需要哈希的分支，声称修复的 sha256 行为在整个测试体系中**零执行**。比"没有测试"更糟：绿灯让"已修"在报告、注释、CHANGELOG 三处落章。规则：**方言/驱动分支的修复，探针必须跑在该分支声称生效的方言上**；r24 的教训是"从未运行的测试面 = 从未验证的声明"，本轮的变体是"**跑在兄弟分支上的绿测试 = 给未接线的修复发假合格证**"。
- **I-2（XOR 只写了一半）**："二选一"在 create 里只查了"至少一个"——缺失拒绝写了，矛盾拒绝没写。互斥约束天然是两半，写其中一半时另一半必须同时决策。
- **I-4（对称面惯性）**：r26 L-5 修了 `StateMachine.clone(deep)`，同一文件族的 `StateTransfer.clone(deep)` 没有进同一张清单——r25 复盘规则 #5（修一侧必把另一侧写进同一矩阵）在 clone 家族上没执行。

## 四、回答根本问题:为什么 27 轮之后仍按轮产出致命 bug

组合空间是指数的（关系类型 × 拓扑 × 载荷形态 × 操作 × 轨道 × 驱动 × 深度 × 扇出……），矩阵覆盖的是人工挑选的切片。既有防御体系的三个支柱——**预言机**（对全体执行的性质断言）、**登记册**（维度的显式决策）、**汇合点**（收敛产生点数量）——分别攻击断言强度、维度自觉、实现面积。27 轮的经验证明它们有效（每轮的新 bug 越来越深、越来越窄），但也暴露了两个仍然开放的结构性缺口：

1. **登记册是反应式的**：每根轴都是被 bug 揭示后回填的（驱动差异轴 ← r24 F-1；消费轨 ← r25 F-1；本轮两根 ← r27 F-1/F-2）。轴的发现依赖 bug 先发生。而轴其实有一个先验来源：**实现的每个分支点就是一根轴**。`Setup.mergeRecords` 的 combined/merged/isolated 分裂、`ComputationSourceMap` 的 per-dep 注册、驱动的方言分支、`NewRecordData` 的递归分类——每处"同一声明、不同处理"的代码分叉都定义了一个等价类分裂。**主动的轴审计 = 走查实现分支点、与登记册求差集**，不需要等 bug。
2. **生成侧仍是人肉的**：预言机已经强到"输入到达即抓获"（F-1 六种形态放进 withEventCompleteness 全部当场红），瓶颈完全在输入形状的生成。人不会生成自己没想到的形状；**随机/属性化生成会**。一个按 schema 随机生成嵌套写载荷（随机拓扑配置 × 随机嵌套深度 × 随机 ref/new/null 混合）、在既有预言机 + 朴素模型对照下运行的 fuzzer，会在数分钟内命中 `profile: { title, teams: [...] }` × reliance-combined。这是把"维度自觉"从必要条件降级为优化手段的唯一路径。

一句话：**缺格的时代靠清单和 sweep 已经基本关闭，缺轴的时代需要把轴的来源从"bug 的尸检"换成"实现分支点的活体检查"，把格子的铺设从"人想到哪铺到哪"换成"生成器铺、预言机判"。**

## 五、本轮已落地的机制改造清单

1. 登记册两根新轴：「载荷嵌套深度 × 子记录拓扑」「单事件 × 多监听扇出」（`WritingComputationTests.md`）；
2. `preprocessSameRowData` 汇合点守卫：combined 子记录的不支持结构全部 fail-fast——「凡被分类的必须被消费或被拒绝」守恒律的追认（六种静默损坏 → 带 workaround 的声明错误）；
3. Scheduler + migration 双消费方的扇出去重（同一契约两处收口，且回归同时铺"该合并的合并"与"不该合并的没被合并"两个方向）；
4. BoolExp 单边 and/or 的四个读者统一左透传语义；
5. `validateAggregationTarget` 汇合点（六聚合类 XOR 两半齐全）；
6. MySQL operationKey 代理键真实接线 + **方言匹配探针**（真实 MySQL 8 上验证 400+ 字符键 round-trip 与共享前缀不碰撞）进 CI；
7. 文档止损：RealTime 调度承诺如实化、storage-layer.mdc 第三元组勘误。

## 六、给后续轮次的操作性规则（增补）

1. **「同一声明、多种物理编译」的特性，按载荷形态 × 编译形态求全积**——一种编译下的绿灯对另一种编译零置信度（F-1）。
2. **凡是「每 X 注册一个 Y」的机制，必须显式回答「多个 Y 同时命中时执行几次」**——集合语义不会从单元素断言中涌现（F-2）。
3. **夹具库必须包含非幂等观察者**（执行计数 spy / useLastValue 累加器）——纯函数夹具对执行次数类 bug 系统性失明（F-2）。
4. **契约面的任何决策（合法化/收紧/废弃）触发与修复同等的读者枚举**，不只是修 bug 时（I-1）。
5. **公开类型面的每个字段必须有行为差异断言**，存在性测试不算；面向 agent 的规则文档与实现的一致性是测试面（I-5）。
6. **方言/驱动分支的修复，探针必须跑在该分支声称生效的方言上**——跑在兄弟分支上的绿测试是假合格证（I-3）。
7. **主动轴审计**：走查实现分支点（Setup 拓扑分裂 / 监听注册 / 方言分支 / 递归分类），与登记册求差集——轴的来源从 bug 尸检前移到活体检查（第四节）。
8. **评估落地属性化生成**：schema 驱动的随机写载荷生成器 + 既有预言机 + 朴素模型对照——把格子铺设从人肉枚举换成机器生成（第四节；建议下轮先做 write-path 一个面的 PoC）。
