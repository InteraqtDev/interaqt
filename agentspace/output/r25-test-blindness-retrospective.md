## 〇、先给结论

r25 F-1 不是新机制的 bug。它的两块拼图在两轮之前就分别被拿在手里:

- **r21 F-1** 把「create 事件 = defaults + payload;缺席的普通值属性 ⟺ 库里 NULL」从口头约定提升为本地 match 求值的**承重假设**(`plainNullableKeys` 按这条契约把缺席键解读为 NULL)。
- **r22 I-4** 在**完全相同的三个产生点**发现了**完全相同的缺陷**(default-only 字段缺席),并把 defaults 补齐修在了 filtered 视图事件轨(`enqueuePostWriteCreationCheck`)。

两轮各自离真相一步之遥:r21 建立契约时没有审计**谁在生产不符合契约的事件**;r22 修复产生点时没有枚举**同一产生点上契约的全部消费轨**(base 名事件 / 视图事件)。于是 base 轨的缺陷带着 r21 的承重假设一起活到 r25——下游不是报错,而是**按契约给出的语义**把缺席解读为 NULL,静默 skip。

这与 r23 复盘命名的形状完全一致:**一个只对第一消费者成立的答案,被默认为对全家成立。** 新增的教训是它的对偶面:**一个被第一消费者依赖的契约,被默认为全体生产者都已履行。**

## 一、显微镜:防线为什么没拦住

### 1.1 契约的两次「接近」都没有闭环

| 轮次 | 拿到了什么 | 缺了哪一步 |
|------|-----------|-----------|
| r16 R-1 | 确立「视图 create 事件 payload = defaults + payload」 | 契约只写进注释与单点测试 |
| r21 F-1 | 本地求值**依赖**「create 事件 = defaults + payload」解读缺席键 | 没有反向审计:哪些生产者违反这条被依赖的契约?(修复的是消费方,假设了生产方健康) |
| r22 I-4 | 发现三个行内产生点漏 defaults,修复**视图轨** | 没有枚举同一产生点的**第二条消费轨**(base 名事件);登记册把「default-only 谓词字段」登记在**视图 × 写形态**行内——把 class 级别的契约写成了 instance 级别的格 |

r22 复盘的标题就是「概念的寄生位置是轴」;r22 I-4 的修复注释也写明「与 base create 事件同构(r16 R-1)」——**知识都在,清单没有强制执行**。「修某个产生点的事件 payload 时,列出该产生点发出的全部事件名并逐一核对契约」这一步,在 r22 是靠人肉自觉,而人肉在「视图事件」这个修复目标上聚焦后,对同一行代码上方五行的 base push 失明。

### 1.2 契约依赖是单向登记的

r21 F-1 修复把快照完备性写进了 `readMatchPath` 的注释与登记册「本地谓词求值 × 快照完备性」行——从**消费方**视角("我可以把缺席解读为 NULL,因为契约保证……")。没有任何机制从**生产方**视角登记("所有 create 事件的产生点必须满足……")。契约的两端只有一端被记录,违反契约的一端恰好在另一端的盲区里。

### 1.3 预言机检查了「有没有事件」,没检查「事件说了什么」

事件完备性预言机(r17 落地)对账的是**事件的存在性**(出现的记录有 create、消失的有 delete、变化字段被 keys 覆盖)。create 事件的 **payload 内容完备性**不在规则集内——一个「存在但缺字段」的 create 事件对预言机是绿的。r22 I-4 的回归测试单点断言了视图事件的 `isPrimary`,但单点断言只保护被断言的那一个字段/那一条轨。

### 1.4 下游测试全部用显式载荷喂 match/trigger 字段

聚合与 StateMachine 的既有测试在建边时几乎总是显式携带 `&` 数据或匹配 payload 直给的字段——「匹配字段仅有默认值 × 行内建边」这个交叉格在 runtime 轨道上是空白。r21 F-1 的四个复现形态覆盖了 computed 列、null、负向操作符、关联路径,唯独没有「default-only ×行内产生点」——因为那被认为是 storage 层已修的问题(r22 I-4),**跨层的格没有人认领**。

## 二、本轮的公共形状(与 F-2 合看)

> **契约/谓词的每个消费位置都是独立的格,且「已修复」标签会让全家提前豁免。**
> F-1:create payload 契约在同一产生点有两条消费轨(base 事件/视图事件),r22 修了一条,另一条被「已修复」的印象掩护了三轮。
> F-2:filtered relation 谓词有三个查询消费位置(attributeQuery 子查询/路径 match/EXIST),r8 做对了第一个、共享别名机制恰好救了第二个,第三个在「filtered relation 查询早就好了」的印象下漏到 r25。

两者都不是无知之错,而是**印象扩散**之错——r23 复盘称之为「第一处修完后印象扩散到全家」,本轮再次字面兑现。

## 三、本轮已落地的机制改造

1. **契约进预言机(把愿望变成机器检查)**:事件完备性预言机新增第 5 条规则——每个 created 行的全部非 NULL 普通值字段必须出现在 create 事件 payload 且值一致(computed/computation 属性经 `createPayloadExemptField` 显式豁免)。今后任何产生点(含未来新增)只要被矩阵套件包裹,payload 缺字段立即红。**规则落地首跑即抓出 combined 嵌套新建 link 事件端点缺 id 的兄弟格**——机制的价值当天自证。
2. **补齐实现收口到单点**:`NewRecordData.completeEventPayloadWithDefaults` 是 defaults 补齐的唯一实现,base 轨(三个产生点)与视图轨(`enqueuePostWriteCreationCheck`)全部经过——两条轨道从此不可能分叉。
3. **登记册升格**:「视图 × 写形态」行把 default-only 格从"视图事件的格"改写为"**create 事件 payload 契约本身是双轨声明面**"(base 轨 r25 F-1 案例入册);「概念的寄生位置」行新增"filtered relation 谓词的三个查询消费位置";新增「驱动差异轴」行(r24 引入、r25 扩展:方言 fieldType 识别 / 连接幂等不变量 / atomic 读写对称)。
4. **沉睡面继续收口**(r24 教训的延伸):CI 从两个 PG 套件扩到全部七个(`npm run test:postgres`);MySQL 连接管理面首次获得真实服务器上的 env-gated 回归(`mysqlOpenIdempotency.spec.ts`)。

## 四、给后续轮次的操作性规则

1. **修事件 payload 时,枚举该产生点发出的全部事件名**(base / 视图 / link / 宿主),每个事件名对同一契约独立核对——「同一行代码附近的兄弟 push」是 instance-fix 残留的高发地。
2. **建立契约时双向登记**:消费方获得的假设(可以把缺席解读为 NULL)与生产方承担的义务(payload 必须含 defaults)要同时写进登记册;只登记单侧的契约等于没有登记——违反者恰好住在另一侧的盲区。
3. **「已修复」不豁免兄弟格**:review 一个概念时,把它的全部寄生位置重新列一遍,逐一问「这个位置的修复证据是什么」——印象不是证据,r8 的 attributeQuery 证据救不了 EXIST。
