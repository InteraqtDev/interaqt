- 日期:2026-07-12
- 关联:`deep-review-2026-07-12-r25.md`(问题发现与修复)、`r17-–r24-test-blindness-retrospective.md`(前几轮复盘)

## 〇、先给结论:六个问题、四类结构性失明

逐案的直接原因各不相同,但归类后只有四种结构性失明——每一种都不是「测试写少了」,而是**测试系统的某类置信度来源本身是虚假的**:

| 类 | 结构性失明 | 本轮案例 |
|----|-----------|---------|
| A | **未被执行的契约**:契约写在注释/登记册散文/类元数据里,没有任何机器执行它 | F-1(payload 契约只存在于注释)、I-4(`static.public` 约束从未接线)、I-1(「方言须识别自家 fieldType」从未被表述) |
| B | **替身弱于本体**:用更便宜的代理物验证,而代理物恰好在关键维度上比本体宽松 | I-1(PGLite 替身比真实 PG 宽容)、I-2(代码目视替代运行时验证)、I-4(测谓词函数替代测执行)、端点 id(partial matcher 替代完整断言) |
| C | **夹具偏置**:手写测试天然回避「我没写的值」与「不该出现的行」 | F-1(default-only 格从不自然出现)、F-2(幻影多行需要诱饵数据才能暴露) |
| D | **特性引入期无交叉扫描**:新轴值落地时不与既有正交特性求积 | F-2(filtered relation × EXIST)、I-1(json 白名单 × 方言入口)、I-3(读侧修复 × 写侧兄弟) |

最刺眼的元事实:**F-1、I-2、I-3 三个问题都紧贴着此前某轮修复自己的边界**——r22 I-4 修了同一缺陷的另一条轨、r22 I-5 的家族 sweep 给 MySQL 发了假合格证、r24 I-1 修了同一 API 的读侧。审查体系不是没看到这些区域,而是**看过之后盖了「已处理」的章,章的覆盖范围被系统性高估**。

## 一、逐案显微镜

### 1.1 F-1(行内 base create 事件缺 defaults):四道防线各自失守

这是本轮最值得解剖的案例,因为它的两块拼图早就分别在手里:

| 轮次 | 拿到了什么 | 缺了哪一步 |
|------|-----------|-----------|
| r16 R-1 | 确立「create 事件 payload = defaults + payload」 | 契约只写进注释与单点测试 |
| r21 F-1 | 本地 match 求值**依赖**该契约解读缺席键(缺席 ⟺ NULL) | 只登记了消费方的假设,没有反向审计生产方——哪些产生点在违反这条被依赖的契约? |
| r22 I-4 | 在**完全相同的三个产生点**发现**完全相同的缺陷**(default-only 字段缺席) | 修复只做在视图轨(`enqueuePostWriteCreationCheck`);同一产生点上方五行的 base 名 push 没有被枚举为第二条消费轨 |

四道本应拦截的防线逐一检视:

1. **事件完备性预言机(r17)检查了「有没有事件」,没检查「事件说了什么」**。规则集覆盖存在性(出现的记录有 create、消失的有 delete、变化字段被 keys 覆盖),create 事件的 payload 内容完备性不在规则内——一个「存在但缺字段」的事件对预言机是绿的。矩阵套件确实包裹了行内写法,但预言机对这个失败模式天生失明。
2. **登记册的格写成了 instance 而非 class**。r22 把该格登记为「行内 link **视图** create 事件的 payload defaults 由 enqueuePostWriteCreationCheck 统一补齐(default-only 谓词字段是独立格)」——签发了一张范围过窄的「已覆盖」签证。后续读者(包括 r23/r24 的审查者)看到 default-only 关键词即认为已收口。**登记册是防兄弟失明的机制,但格的措辞如果编码了 instance,它反而会放大失明**。
3. **契约依赖是单向登记的**。r21 把「可以把缺席解读为 NULL」写进消费方(`readMatchPath` 注释与「本地谓词求值 × 快照完备性」行),没有任何机制从生产方登记对偶义务(「所有 create 事件产生点必须带 defaults」)。契约两端只记了一端,违反者恰好住在另一端的盲区。
4. **下游测试全部用显式载荷喂 match/trigger 字段**。聚合与 StateMachine 的既有测试建边时总是显式携带 `&` 数据或匹配 payload 直给的字段——写测试的人自然会先写入自己要断言的值。「匹配字段仅有默认值」是**「我没写它」的退化格**,手写夹具系统性地回避它。r21 F-1 的四个复现形态(computed 列/null/负向操作符/关联路径)恰恰漏了这一个,因为它被认为是 storage 层已修的问题——**跨层的格没有人认领**。

### 1.2 F-2(filtered relation × EXIST):正确性靠巧合,巧合无法被审计

1. **路径 match 位置的正确性是「机制副产品」,从未被文档化为承重结构**。`activePosts.title` 之所以正确,是因为 rebased 谓词原子与路径原子共享同一条 JOIN 路径、被查询树合并到同一别名——「同一条边」语义由别名合并**碰巧**保证。没有人写下「这里的正确性依赖别名合并」,于是没有人能问出「EXIST 有没有同样的保护?」**当正确性依赖一个未被陈述的机制时,兄弟位置无从审计**。
2. **EXIST 测试先于 filtered relation 存在,特性落地时没有回填扫描**。EXIST 的测试(existQueryProof、r19 F-2 引用收集)全部建立在 base 关系上;filtered relation(r8+)引入时没有「新特性 × 既有查询形态全集」的求积步骤。r22 复盘已把 filtered *entity* 的四个寄生位置立轴,但 filtered *relation 的谓词*的消费位置从未被枚举。
3. **正向断言偏置**。既有 filtered relation 查询测试断言「满足谓词的边能查到」;幻影**多**行的失败模式需要**诱饵构造**——一条不该命中的边 + 另一条让外层 AND 为真的边。登记册里「对抗性断言」轴(r19)只覆盖权限守卫的 fail-open,没有推广到查询语义的 fail-open(多返回行本质上就是查询面的 fail-open)。

### 1.3 I-1(type:'json' 方言分裂):白名单扩张无消费方扫描 + 替身比本体宽容

1. **r23 扩白名单时没有消费方扫描**。把 `'json'` 加入 Property.type 白名单扩大了声明面,但没有任何流程强制「白名单每加一个值,扫一遍该值的全部消费方」——`mapToDBFieldType`(四驱动)、方言入口守卫(四驱动)、读归一化、匹配编译。声明白名单与驱动方言入口相隔四个文件、两个层,靠人肉联想不可能稳定命中。
2. **PGLite 作为 PostgreSQL 替身,恰好在关键维度上更宽容**。PGLite 的方言守卫是 `toLowerCase()`,真实 PG 是 `===`——替身通过的声明在本体上裸报错。**替身的安全方向应当是「至少与本体一样严格」,这里反了**。r24 复盘已确认「PGLite ≈ PG 只在 SQL 方言层成立」,但驱动差异轴当时只登记了 id 分配机制,没有推广到「方言入口对 fieldType 字符串的识别」。
3. **一个零成本的自洽契约从未被写下**:「每个驱动的 parseMatchExpression 必须识别自己 mapToDBFieldType 产出的全部 json 形态」是纯函数对的性质,不需要任何数据库服务器就能测——但因为它从未被**表述**为契约,也就从未被测试。本轮已落地(见第三节 #3),旧驱动上红测恰好在缺陷处失败。

### 1.4 I-2(MySQL open 泄漏):家族 sweep 做了,但用代码目视发了假合格证

这一案的教训最精确,值得原文引用。r22 I-5(SQLite open 幂等)的报告写道:

> PG(`if (this.pool)`)/**MySQL(`if (this.db) return`)都有幂等守卫**,SQLite 是四驱动中唯一漏网。

**r22 确实执行了 fix-the-class 清单的家族枚举**——但 MySQL 的 `if (this.db) return` 守卫在 `openForSchemaRead()` 上,`open()` 上没有。目视扫描验证到了方法对里错误的那一个成员,然后在报告里给 MySQL 盖了「已有守卫」的章。这枚假合格证的效力持续了三轮:后续审查者(含 r24 审计 env-gated 面时)读到这句话就把 MySQL 从怀疑名单上划掉了。

三个叠加因素:

1. **枚举 sweep 的证据等级没有要求**。清单说「枚举同一声明面的全部读者并逐一验证」,但「验证」被默许为读代码。**代码目视是运行时验证的弱替身**——尤其对连接生命周期这种状态机,眼睛会把两个相邻方法看成一个。
2. **零覆盖面不产生任何可见信号**。PG 套件缺环境时至少显示 `skipped` 计数(r24 由此警觉);MySQL 连套件都不存在,测试报告里没有任何标记提醒「这四百行驱动代码从未在真实服务器上执行过」。**缺席的覆盖比跳过的覆盖更隐形**。
3. **泄漏类失败对功能断言不可见**。即使有 MySQL 功能测试,连接泄漏也不会让任何 expect 失败——它需要专门的计数预言机(本轮的回归即以 `information_schema.processlist` 计数断言)。

### 1.5 I-3(atomic 写路径 json):矩阵有一个象限没铺,bug 恰好住在那

r24 I-1 的回归测试(`review-fixes-2026-07-12-r24.spec.ts`)实测覆盖了:record-target **get**(boolean+json)、record-target **replace**(仅 boolean)、global boolean。也就是说 **(操作 × 值类型) 矩阵里恰好 replace/CAS × json 这个象限没铺**——写侧 json 正是崩溃处。两个结构性原因:

1. **「读/写对称」不是登记册轴**。r24 修读侧时,「同一 API 的写侧是否有对称缺口」不在任何强制清单上;修复者的注意力聚焦在「读出来的类型与 find 一致」,写进去的参数形态没有进入视野。
2. **公开 API 面只靠框架内部用法提供顺带覆盖**。atomic 的框架内消费方(RecordBoundState/ActivityCall)全部是 number/string/boolean——json 象限没有任何内部用户,于是集成测试的顺带覆盖恒为零。**公开面必须靠刻意铺的矩阵,不能指望顺带**;`atomicState.spec.ts` sanction 了业务列用法却没有铺满类型轴。

### 1.6 I-4(merged 守卫未接线):测试了刀刃,没测试铡刀

`tests/core/entityRelation.spec.ts` 有一个测试名为 *"inputRelations.constraints.mergedRelationNoProperties rejects properties on merged"*——它**直接调用约束谓词函数**并断言返回 false:

```ts
const constraint = Relation.public.inputRelations.constraints.mergedRelationNoProperties;
expect(constraint({ inputRelations: [r1], properties: [...] } as any)).toBe(false);
```

谓词逻辑是对的、测试是绿的——但 `create()` 从未调用这个谓词。**测试验证了机制的部件,没有验证部件被装进机器**;绿灯提供的是关于一台没接线的机器的置信度。这是「替身弱于本体」的第四种形态:测函数 ≠ 测执行。

根因是 createClass 债(r16 建议 4,已八轮复确):校验靠每类每字段手工接线,「声明了约束 ⇒ 约束被执行」不是系统不变量,每个新声明面都会重新制造这个缺口。只要统一校验不落地,同类缺口会继续按轮产出(本轮 I-4 自己又添了四处手写守卫)。

### 1.7 附带(combined link 事件端点缺 id):partial matcher 把 bug 写进了绿灯

`oneToOne.spec.ts` 对该 link 事件的既有断言是 `source: { title: "f2" }`——**partial matcher 断言了字段却没断言 id**,而 id 恰好是缺的那个。旧测试的期望形状与 bug 的形状完全重合(r19 复盘「测试不是漏写,是替 bug 挡住了所有想修它的人」的又一实例,只是这次以欠断言而非错断言的形式)。它在预言机第 5 条规则落地的**首跑**即被抓出——同一天内证明了「机器检查的契约」与「人手写的 partial 断言」之间的置信度差距。

## 二、公共形状:为什么「第二消费方」被点名三轮后仍然复发

r22 复盘立了「概念的寄生位置是轴」,r23 复盘立了「成功语义歧义/契约第二消费方」,r24 复盘立了「从未运行的测试面 = 从未验证的声明」。**本轮六个问题里五个仍然是这些已命名形状的实例**。为什么命名不阻止复发?

因为既有防御是**修复时刻的人肉清单**:「枚举同一声明面的全部读者」的枚举集合由修复者的心智模型定义,而心智模型恰恰被当前修复目标(视图事件/读路径/openForSchemaRead)锚定。清单无法命令一个人看见他没想到的东西。

回看历史,真正阻止过复发的只有三种**机械形态**:

1. **汇合点**(一份实现,消费方无从分叉)——r22 F-1 的 `resolveBaseRecordName`、本轮的 `completeEventPayloadWithDefaults`;
2. **预言机**(对全体实例的性质断言,不依赖修复者枚举)——事件完备性预言机、本轮的 payload 规则(落地当天抓出人没枚举到的端点格);
3. **带运行时证据的逐格 sweep**(每格一个真实执行的探针,目视不算数)——r24 引入真实 PG 首跑抓 F-1,本轮引入真实 MySQL 首跑抓 I-2、方言契约测试红在旧驱动上。

而**散文形态的知识(登记册措辞、报告结论、修复注释)不仅不阻止复发,还可能发放假签证**:r22 报告里「MySQL 有幂等守卫」一句话、登记册里「视图 create 事件的 defaults 已统一补齐」一格,都在后续轮次里充当了「此处已安全」的错误信号。**登记册是双刃的:格写窄了,它就从瞄准镜变成遮光板。**

## 三、本轮已落地的机制改造(全部有红-绿证据)

1. **契约进预言机**:事件完备性预言机新增第 5 条规则——每个 created 行的全部非 NULL 普通值字段必须出现在 create 事件 payload 且值一致(computed/computation 属性经 `createPayloadExemptField` 显式豁免)。首跑即抓出 combined 端点缺 id 的兄弟格。(A 类失明的机制回应)
2. **补齐实现收口单点**:`NewRecordData.completeEventPayloadWithDefaults` 是 defaults 补齐唯一实现,base 轨与视图轨从此不可能分叉。(「第二消费方」的汇合点回应)
3. **驱动方言自洽契约测试**(`tests/storage/driverDialectConsistency.spec.ts`,本复盘新增):四驱动 × 三种 json 声明形态 × 四操作符逐格断言「同 json 语义的 fieldType 形态必须得到完全相同的方言处理」,且 PG 系/MySQL 必须接管回退路径无法表达的操作符。纯函数对,无需服务器,旧驱动上红测恰在 I-1 缺陷处失败(PG/MySQL 的 'json' vs 'JSON' 分裂 + contains 未接管)。(B 类「替身宽容」与 D 类「白名单扩张」的永久守卫——未来任何 mapToDBFieldType 产出形态的扩张都被自动对账)
4. **MySQL 面首次获得运行时覆盖**:`mysqlOpenIdempotency.spec.ts`(env-gated,连接计数预言机)——把 r22 的假合格证换成可重跑的运行时证据;红测确认旧代码同序列悬挂 4 条连接。
5. **沉睡面收口**:CI 从 2 个 PG 套件扩到全部 7 个(`npm run test:postgres`);建议后续补 MySQL service container 让 #4 常驻运行。
6. **登记册升格**:「视图 × 写形态」行把 default-only 格改写为「create 事件 payload 契约本身是双轨声明面」;「概念的寄生位置」行新增 filtered relation 谓词的三个查询消费位置;新增「驱动差异轴」行(方言 fieldType 识别 / 连接幂等不变量 / atomic 读写对称 / 环境可得性)。

## 四、仍然开放的结构性缺口(诚实清单)

1. **createClass 统一声明期校验**(r16 建议 4,九轮)——I-4 类缺口的工厂。只要校验靠手工接线,「声明的约束不被执行」就会按轮再生。这是当前测试系统无法自愈的唯一一类:约束「已声明」这个事实本身就在制造虚假置信度。
2. **atomic API 面矩阵未铺满**:本轮只补了 json 写象限;(get/replace/CAS/increment/lock)×(number/string/boolean/json)×(4 驱动) 的完整矩阵仍未刻意铺设。
3. **timestamp 读归一化格**(boolean/json 已收口后的最后一格)——行为变更,需 CHANGELOG 配套决策。
4. **「特性引入期交叉扫描」还没有成为流程**:新白名单值/新声明形态落地时,against 登记册全部既有行做一次求积检查,应写入开发工作流(而不是等下一轮 review 撞见)。
5. **代理强度审计**:所有「替身」(PGLite 替真 PG、partial matcher 替完整断言、目视替运行时)应逐一回答「替身在被断言的性质上是否至少与本体一样严格」——目前这只是本复盘的一条散文建议,尚无机制形态。

## 五、给后续轮次的操作性规则(修订版)

1. **sweep 的证据等级**:fix-the-class 的家族枚举,每格必须有**运行时探针**证据(测试或可重跑脚本);代码目视结论一律标注「未运行验证」,不得写成「已有守卫」这类合格证语言。(I-2 教训)
2. **契约双向登记**:任何「消费方可以假设 X」的契约,必须同时登记「生产方义务 ⊢ X」并优先落成预言机规则;只登记单侧的契约视为未登记。(F-1 教训)
3. **正确性依赖的机制必须点名**:如果某位置的正确性依赖一个非显式机制(别名合并、引用共享、执行顺序),把机制写进注释并把**不具备该机制的兄弟位置**列为可疑格。(F-2 教训)
4. **白名单/枚举面扩张即扫描**:扩一个值,列出该值的全部消费方(跨层、跨驱动)并逐一验证——理想形态是像方言自洽契约那样,写一个对「形态全集」量化的测试,让未来扩张自动被对账。(I-1 教训)
5. **修 API 的一侧时,把另一侧写进同一张矩阵**:读/写、编码/解码、加锁/解锁——对称面是「第二消费方」在 API 形态上的特例。(I-3 教训)
6. **事件 payload 断言优先走预言机**,手写 partial matcher 只用于补充语义断言,不作为 payload 完备性的置信度来源。(端点 id 教训)
