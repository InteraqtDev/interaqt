# 深度反思：为什么已有的测试体系此前没有抓到 r30 的这些致命 bug

- 日期：2026-07-15
- 关联：`deep-review-2026-07-15-r30.md`、`r28-test-blindness-retrospective.md`、`quality-foundation-plan-r27.md`
- 性质：测试与修复体系的结构性复盘。本轮特殊性：四个致命项分别逃过了**四种不同的防线**——
  生成式 fuzzer（A）、手写异步套件 + r2 并发加固（B）、活动层手写测试 + 历轮 fail-fast 群（D2）、
  迁移生成 fuzzer + 手写迁移套件（E）。每一个都不是"没写测试"，而是**防线的辖区声明与
  实际可达空间存在从未被显式对账的差集**。

---

## 〇、先给结论：四个 bug 的一句话逃逸机理

| 编号 | 逃逸机理（一句话） | 防线失明的类型 |
|------|------------------|--------------|
| A（filtered×combined 剪枝误删） | 形状**在生成域里**，但没有任何预言机**读过那个面**（经 filtered 属性名的嵌套读取） | 预言机读取面缺格 |
| B（async 陈旧覆写） | "最新性"不变量实现在**代理对象**（task 行）上，绕过代理的写者（同步路径）对它不可见 | 不变量载体错位 |
| D2（跨分支 transfer 污染） | 解析器（全图共享的 rawToNode）服务的空间比校验器（本层 candidate 集合）检查的空间**宽** | 解析器辖区 ≠ 校验器辖区 |
| E（迁移链 delete 死路） | 手写测试把「失败一次→dryRun→手工补 id」的 workaround 舞步**固化成了预期行为**；审计的静态性假设从未被写下 | 测试为 bug 背书 + 假设未登记 |

**共同结构**：r28 复盘立了「探针辖区 = 结论辖区」的规则（探针验证过的形态边界必须显式标注）。
r30 的四个 bug 证明这条规则需要**全称化**：不只探针——预言机、不变量、校验器、审计，
**每一道防线都有一个隐式的辖区声明**（读哪些面 / 载在哪个对象上 / 检查哪个键空间 / 假设什么静态性），
而辖区从未被要求显式写出并与可达空间求差集。四个 bug 全部住在这个差集里。

## 一、复盘自身的证据纪律：一次归因修正的记录

本复盘对每个"为什么没抓到"都做了证伪实验，其中 A 的实验**推翻了 r30 报告初版的归因**，
值得完整记录——错误的逃逸归因会导向错误的机制建设：

- **初版归因**（r30 报告 §六初稿）：「r29 把 EXT-1 家族从 CI 生成域移除，顺带放走了 A」。
  这个归因**是错的**：EXT-1 的生成域限制作用在 **merged (union) entity** 的关系端点上
  （`fuzzSchema.ts` 的 `excluded` 集合只影响 `mergedEntities` 的候选池）；A 的形状是
  **filtered relation over combined base**，属于 `filteredRelations` 生成路径——
  `withWeight = relationChoices.filter(c => c.linkProps.includes('weight') && !c.symmetric)`
  的菜单里**包含** `1:1-reliance`（combined）关系，`chance(0.7)` 抽中即生成。
- **证伪实验**：把 A 的修复回退到 r28 后代码，给 fuzzer 新增「filtered 嵌套读取完备性」
  预言机（第 7b 条，见 §六），跑 extended 种子 1–40：**seed 2 当场变红**，红点正是 A 的
  损坏面（`FzS2_FR: pairing 2->2 visible on the link face but MISSING from the
  FzS2_B.fr_out2 nested read`）。恢复修复后 1–120 全绿。
- **修正后的结论**：形状自 r29 起就被生成了几十次（extended 池 70% 概率含 filtered relation，
  关系菜单七分之三是 1:1 家族），**每一次都带着 bug 跑完了全部 8 条预言机并拿到绿灯**。
  失明的不是生成器，是预言机的**读取面**：
  - 第 6 条（配对读取一致性）遍历 `schema.relationChoices`——只读 **base 属性面**
    （`choice.sourceProperty`），从不读 filtered 属性名（`fr_*`）；
  - 第 7 条（filtered 谓词一致性）读 `findRelationByName(filteredName)`——**relation-name 面**，
    而这个面恰好一直是正确的（link id 真相源）；
  - **没有任何一条规则经 filtered 属性名做实体嵌套读取**——A 的损坏恰好只住在这个面上。

这个修正本身就是教训：**逃逸归因必须做敏感性实验**（在 buggy 代码上验证"补上的机制会红"），
否则复盘会把机制建设预算花在错误的缺口上（按初版归因，我们会去"回补生成域"——
而生成域根本没缺，补了也还是绿）。

## 二、A 的法医检视：覆盖 = 生成域 × 预言机读取面（乘积，不是并集）

### 2.1 防线逐一检视

| 防线 | 为什么没响 |
|------|-----------|
| 手写 filtered relation 套件（`filteredRelation.spec.ts`，10 个用例、2000+ 行） | 全部用例的 base 关系是 **1:n / n:n**（"active posts"式的集合过滤是 filtered relation 的动机场景）。x:n 的嵌套读取走 `findXToManyRelatedRecords` 独立查询（谓词作为 WHERE 正确生效），**根本不经过出 bug 的 combined 同行读取路径**。声明面允许 `baseRelation` 是任何关系（含 1:1 reliance→combined），但测试宇宙只有动机形状——r27 命名的**夹具偏置**，这次偏置的轴是 base 关系的**基数/物理拓扑** |
| r28 prune 修复自带的回归测试（18 个用例） | 全部铺在**动机面**（combined 直接读、幻影同住、多 owner reliance）。r28 执行了 fix-the-class 清单第 1 条（枚举 combined 读取的全部消费点：嵌套读取/match 路径/删除级联/flashOut），但枚举的是「**事实的消费点**」，没有枚举「**读路径的视图形态**」——filtered relation 是读路径的视图乘子，登记册 r22 的「概念寄生位置」轴早已声明"每个寄生位置都是独立的格"，但 r28 修复时没有把"filtered 视图"当作 combined 读路径的一个寄生位置来扫 |
| fuzzer（extended 模式，r29 起含 filtered relation） | **生成了、跑了、绿了**——见 §一。8 条预言机没有一条读 filtered 属性的实体嵌套面 |
| 登记册「监听名形态」轴（base 名/filtered 名/merged input 名） | 该轴只被应用在**生成侧**（fuzzOps 有概率经 filtered 名写入）与**监听侧**（死监听不变量），从未应用在**预言机的读取侧**。轴存在，但轴的消费方不完整——这是登记册机制自身的盲点：轴入册时没有机械枚举"这根轴要作用于哪些机制面（生成/监听/读取/对账）" |

### 2.2 更深一层：同一次修复内部诞生的"读者对分叉"

r28 的 prune 修复自身就制造了两个拓扑读者：`AttributeQuery` 的 synthetic-`&` 判定
（读 filtered link 的 `mergedTo`——恒非 combined）与 `pruneUnpairedCombinedReads` 的
isCombined 判定（读已解析为 base 的 attributeName——combined）。fix-the-class 清单的
"枚举全部读者"针对的是**修复前已存在的读者**；没有一条规则问：「**我这次修复自己
新增了几个读者？它们读的是同一个投影吗？**」——两个读者在同一个 PR 里诞生、
在同一个 PR 里分叉。见 §八规则 2。

## 三、B 的法医检视：不变量的载体错位 + 单路径夹具想象力

### 3.1 "最新写入胜出"不变量长在了代理对象上

异步计算的正确性不变量是：**同一 freshnessKey 上，最后一次产出的值胜出**。
被保护的对象是「值」；但实现把不变量载在了「task 行」上（`isLatestAsyncTask` 按行 id 排序）。
task 行是产出轨的**代理**——只有 async 轨会创建它。sync/resolved 轨产出值但不留代理痕迹，
于是对代理上的不变量**不可见**：旧 task 在它自己的宇宙里永远是"最新的"。

r2 修 TOCTOU 时加固的是**并发的 task 完成者之间**的竞争（锁 freshnessKey 全组行再判最新）——
写者枚举只枚举了"经过代理的写者"。**不变量载在代理上时，写者枚举天然漏掉绕过代理的写者**；
这不是枚举不认真，是载体选择决定了枚举的可见范围。正确的问题是：
「这个不变量保护的对象是什么？所有能写该对象的轨道（不只是所有能写代理的轨道）是哪些？」
——sync 轨、resolved 轨、迁移回填轨（`writeComputationResult` 也直写值）全部浮现。

### 3.2 为什么手写 async 套件全绿

`asyncComputed.spec.ts` 的结构：一个用例测纯 async 流（create→task→复完→apply），
同一用例末尾补一个纯 resolved 流（**另一条记录** url2）。两条路径**从未在同一条记录
（同一 freshnessKey）上交错**。这是手写夹具的固有想象力边界：每个用例讲一个单线故事。
r28 复盘把「损坏前置状态的最短装配步数」登记为轴（状态历史深度）；B 揭示这根轴在
计算层的投影是**返回形态的交错序列**——async→sync→旧 task 完成，三步、两种返回形态、
一个 freshnessKey。计算层生成式 fuzz（r29）明确把 async 计算列为未纳入域（§1.5），
所以机器侧同样为零覆盖。轴已入册（见 §七）。

## 四、D2 的法医检视：解析器辖区 ≠ 校验器辖区，与"搁置项"的风险等级错判

### 4.1 结构性机理

`buildGraph` 的两个组件对"图的节点空间"持不同辖区：

- **解析器**（`rawToNode`/`uuidToNode`）：整条递归**共享**的实例状态——能解析出任何层级的节点，
  包括 group 子活动内部的；
- **校验器**（candidateStart/End 与 start/end 计数）：**本层**的 interactions/groups 集合。

父级 transfer 引用嵌套节点时：解析器愉快返回节点、校验器的 `delete` 对不在集合里的元素
是 no-op（Set.delete 的静默语义！）、单步嵌套源的 `next===null` 让"同源多 transfer"守卫
也不触发——三道防线各自按自己的辖区看，谁都没错，组合起来是静默接受。
这与 r28 复盘 §1.2 的「守卫的键空间比不变量的键空间窄」（r23 F-1 的教训）是**同一个形状**：
不变量是"transfer 只能连接本层节点"，守验落点却锚定在"start/end 计数=1"这个影子上。

### 4.2 这是一个已知家族的搁置项——搁置流程缺一次风险再评估

r28 报告 §四.4 已经记录了同家族的兄弟形态（「父级 Transfer 指向嵌套子活动交互时孤儿化
ActivityGroup，需要先做产品决策再收紧」）。搁置的理由是"跨层 jump 是否要支持是产品决策"。
但搁置时没有问：**在决策悬置期间，该形态的可达后果是 fail-loud 还是静默损坏？**
r30 证明存在**主动损坏面**（不只孤儿化——跨分支改写 next 指针、every/any/race 语义静默错乱）。
产品决策可以慢，**静默损坏面的临时 fail-fast 不能等**：先拒绝、决策后再放开，
与 Gateway / program group 的既有处置完全同构。搁置清单需要一列「悬置期间的最坏后果等级」，
静默级的搁置项必须带临时守卫。

### 4.3 为什么没有任何测试触发

活动层不在任何生成器辖区（r29 §1.5 明列）；手写活动测试只构造有意义的图——
"transfer 指向别的分支内部"在人的心智模型里是"没道理的写法"，而 taboo 形状恰恰是
生成器无禁忌覆盖、手写永不覆盖的地方（r27 夹具偏置的第三次实证：r28 双 combined、
r29 merged 域、r30 活动图）。活动图的声明空间小（节点/transfer/group 的组合），
是**声明期随机图生成**的理想目标——列入 fuzzer 扩张路线（§七）。

## 五、E 的法医检视：测试为 workaround 背书 + 未登记的静态性假设

### 5.1 一个比"缺测试"更糟的形态：测试把坏流程固化成规范

`migration.spec.ts` 的「changed Transform output requires destructive opt-in」用例的
arrange 段是这样写的：

1. `await expect(migrateWithApproval(...)).rejects.toThrow(...)` ——**断言第一次迁移必然失败**；
2. 失败之后重新生成 diff（此时 storage.queryHandle 已被失败的尝试初始化——**依赖失败的副作用**）；
3. `dryRun` 一次取出 `deletionScope`；
4. **手工把 scope ids 补进 decision** 再migrate。

这四步舞是「fresh controller 在 diff 期发现不了 Transform stale ids」这个缺陷的 workaround，
但它以绿色测试的形态存在于套件里——**从证据面看，它和"设计如此"无法区分**。
r19 F-1 的教训是"错误断言替 bug 挡住了修它的人"；E 是同一教训的**流程版**：
不是断言错了，是 arrange 段的仪式性扭曲被当成了正常使用方式。任何一审查者看到这个测试
都会得出"迁移收缩场景已覆盖、工作正常"的结论——它主动制造了已覆盖的假象。
判据（进 §八规则 5）：**一等工作流的测试如果需要"先失败一次/依赖失败副作用/手工回填
机器可算的数据"才能通过，它就是一份 bug 报告，不是一份规范**。

### 5.2 审计模型的静态性假设从未被写下

`getDestructiveDeletionScope` 对 rebuildPlan 里的每个计算**独立**地按**当前数据**求
"将被删除的 id 集合"。这隐含一个假设：**每个计算的破坏范围可以在迁移执行前静态求出**。
链式 Transform 恰好破坏它——依赖计算的范围取决于上游**迁移后**的状态（不动点/顺序问题）。
假设从未被写成注释或断言，于是：写 `writeComputationPatch` 无条件拒绝 delete 的人
按"审计会提前发现一切删除"行事，写审计的人按"每个计算独立求值就够了"行事，
两个组件对同一假设持不同理解，组合成死路。迁移 fuzzer 的加法限定（r29 §1.5 诚实边界）
使机器侧也无覆盖——但值得注意：**kill-resume 预言机本可以抓住它**（resume 永不收敛
正是预言机的红色形态），缺的只是把破坏性变异放进生成域。

## 六、共同结构：防线辖区的显式化（r28 规则的全称化）

把四个案例放在一起，逃逸空间的统一表述是：

> **每道防线（预言机/不变量/校验器/审计/探针）都有一个辖区
> ——它读哪些面、载在哪个对象上、检查哪个键空间、假设什么前置条件——
> 辖区几乎总是隐式的；bug 住在「隐式辖区」与「可达空间」的差集里。**

r28 复盘的「探针辖区 = 结论辖区」只约束了探针。r30 的四个实例把它推广到全部防线：

| 防线 | 辖区维度 | r30 实例 |
|------|---------|---------|
| 预言机 | **读取面**（读哪些名字形态 × 哪些 API 面） | A：8 条规则没有一条经 filtered 属性名做嵌套读取 |
| 不变量 | **载体**（长在被保护对象上还是代理上） | B：最新性载在 task 行上，sync 写者绕过 |
| 校验器 | **键空间**（检查的集合 vs 解析器服务的集合） | D2：本层集合 vs 全图 map |
| 审计 | **前置假设**（静态可求？顺序无关？） | E：破坏范围的静态性假设被级联破坏 |
| 手写测试 | **叙事形态**（单路径故事 / workaround 背书） | B 的单线夹具、E 的四步舞 |

对应的机制化落点：r29 已确立「fuzzer 的强度 = 预言机的强度 × 生成器的表达域」；
r30 补上第三个因子——**预言机的强度不是标量，是「规则 × 读取面」的矩阵**。
诚实边界（每个 fuzzer 的 §1.5 式声明）必须双侧登记：生成侧缺什么形状、**观察侧缺什么面**；
且诚实边界清单要按「缺口内的 bug 能否静默」排序成工作队列——静默级缺口要么补域/补面，
要么加临时 fail-fast（D2 的处置样板）。

## 七、对防御体系本身的结论（本轮成绩单）

| 侦测机制 | 本轮交付 | 失手处与修正 |
|----------|---------|------------|
| fuzzer（storage 结构化，extended 模式） | （事后）新增第 7b 条后 seed 2 一击命中 A | 生成域包含 A 的形状却零预言机读 filtered 属性嵌套面——**观察侧盲区**，本轮补第 7b 条并做敏感性自检（buggy 红 / fixed 120 种子绿） |
| 计算层生成式 fuzz | — | async 计算不在域内（r29 已声明），B 类交错序列零覆盖——列为下一步扩张点（返回形态交错轴） |
| 迁移生成式 fuzz | — | 加法限定（r29 已声明），E 的破坏性链式变异零覆盖；kill-resume 预言机已具备判定力，缺的是生成域 |
| 手写特性测试（2100+） | 防倒退全绿 | B：单路径夹具；E：**为 workaround 背书的绿测试**（比缺失更糟，主动制造已覆盖假象） |
| fix-the-class 清单 | r30 四个修复的兄弟面扫描 | A 揭示清单第 1 条的缺口：枚举"事实的消费点"没有枚举"读路径的视图形态"；且没有规则约束"修复自身新增的读者对" |
| 主动轴审计 / 基线对照 | A（对照 r27 前基线定谳回归）、B/D2/E（未覆盖域定向探查）全部由人工审查抓获 | 与 r28 结论一致：手写用例已不产出新发现，机器化机制各有辖区且不重叠——本轮全部四个致命项都落在三个机器化机制的辖区**之外**，人工审查仍是辖区差集的唯一覆盖者 |

## 八、已落地的机制改造清单（本轮）

1. **预言机第 7b 条：filtered relation 嵌套读取完备性**（`writePathStructuralFuzz.spec.ts`）——
   link 面（`findRelationByName(filteredName)`）可见的每一条配对必须出现在经 filtered 属性名的
   实体嵌套读取里。敏感性自检：未修复代码 seed 2 红、修复后 extended 1–120 × 30 全绿。

## 九、给后续轮次的操作性规则（增补）

1. **辖区显式化（全称版）**：每新增/修改一道防线（预言机规则、不变量、校验器、审计、探针），
   必须写明其辖区——读取面、载体、键空间、前置假设——并与可达空间求一次差集；
   「诚实边界」段落必须双侧登记（生成侧形状 + 观察侧面）。
2. **修复自查新增读者对**：fix-the-class 清单第 1 条补充——枚举"我这次修复自己引入了哪些
   新读者/新判定点"，并验证它们读同一投影（A：synthetic-`&` 与 prune 在同一 PR 里诞生并分叉）。
3. **不变量落在被保护对象的维度上**：为"X 胜出/X 唯一/X 守恒"类不变量选择载体时，
   载体必须覆盖**所有能写 X 的轨道**；载在代理上时，逐轨枚举"哪些写者绕过代理"（B）。
4. **搁置项带最坏后果等级**：任何"待产品决策"的搁置项必须标注悬置期间的最坏可达后果；
   静默损坏级的搁置项立即加临时 fail-fast，决策后再放开（D2 vs r28 §四.4 的教训）。
5. **workaround 形态的测试是 bug 报告**：一等工作流的测试若需要"预期失败一次/依赖失败副作用/
   手工回填机器可算的数据"才能通过，按 bug 处理而不是按规范维护（E 的四步舞）。
6. **逃逸归因必须做敏感性实验**：复盘声称"补上机制 M 就能抓住"之前，必须在 buggy 代码上
   验证 M 变红（本轮 A 的归因修正正是靠这一步发现初版归因错误）。
7. **读取面是名字形态轴的强制消费方**：任何按名字形态（base/filtered/merged-input）分身的概念，
   其**全部读取 API 面**（实体嵌套读、relation-name 读、match 路径、计算 dataDep 读）
   都是该轴的格；预言机/回归对每个面逐一铺点或显式豁免。
